# 最小项目状态契约

> 治理状态：已确认  
> 证据状态：独立复核  
> 说明：最小上下文工作集和按需恢复规则已完成最终限域复核  
> 日期：2026-06-11  
> 目的：支持 MVP 中的新线程状态重建，不承诺完整恢复所有历史对话

## 1. 能力边界

MVP 支持“最小状态重建”：

- 从项目文件恢复当前目标、阶段、有效决定、下一步和风险。
- 发现文件缺失、过期或冲突时明确报告。
- 不依赖聊天记录作为唯一事实来源。
- 不要求新线程加载全部项目治理文档；先恢复最小工作集，再按任务需要读取。

MVP 不保证：

- 恢复所有历史讨论细节。
- 自动判断所有过期信息。
- 在文件未及时更新时无损恢复。
- 替代版本控制、备份或外部项目管理系统。

完整线程恢复、自动冲突检测和历史追踪属于 MVP 后能力。

## 2. 最小文件

预计跨线程或需要恢复时，MVP 默认只维护：

```text
docs/
  PLAN.md
```

按需增加：

```text
docs/
  PRD.md          # 产品范围或独立验收标准需要权威基线时
  DECISIONS.md    # 存在需要跨线程保留的已确认决定时
  audits/         # 声明待确认候选或已确认基线时
```

不为快速任务创建这些文件。里程碑数量本身不触发 PRD、DECISIONS 或审计文件；只有实际恢复需求、权威边界或用户明确要求才创建。

## 3. 必需状态字段

紧凑 `docs/PLAN.md` 只要求：

```markdown
# 实施计划

> 执行状态：草案 | 已确认 | 执行中 | 已完成 | 已废弃
> 更新时间：YYYY-MM-DD HH:mm Z
> 当前阶段：阶段名称

## 当前快照
- 目标
- 关键约束（如有）
- 已完成
- 下一步
- 阻塞（如有）
```

只有适用时增加以下字段或章节：

- `最新审计`、`预登记审计`：只有候选或确认事务使用。
- `当前实现契约`：只有正式契约已触发时使用。
- `需要用户决定`、`当前风险`、`相关有效决定`：存在对应内容时使用。
- `当前工作模式`：模式本身影响恢复后的执行时使用。

存在 `docs/PRD.md` 时至少提供：

- 当前有效目标。
- 非目标。
- MVP 范围。
- 关键验收标准。
- 需求条目的治理状态、文档证据状态和更新时间。

存在 `docs/DECISIONS.md` 时，每个决定至少提供：

- 唯一编号。
- 治理状态：建议确认、已确认、已废弃或已替代。
- 证据状态：未检查、已自查、独立复核或已验证。
- 决定内容。
- 原因。
- 适用范围；允许继承文档级默认值。
- 日期；允许继承文档级默认值。
- 替代关系（如适用）。

治理状态表示用户是否接受决定，证据状态表示决定经过了何种验证。二者不能互相替代。

ADR 替代关系只允许以下固定行；缺少某一方向的行表示该方向为空：

```text
- 替代关系：无
- 替代关系：replaces ADR-001, ADR-002
- 替代关系：replaced-by ADR-003
```

`replaces` 后允许一个或多个按编号升序、逗号加空格分隔的旧 ADR ID；`replaced-by` 只允许一个新 ADR ID。同一 ADR 最多各出现一行 `replaces` 和 `replaced-by`，因此一个决定可以替代旧决定后又被更新决定替代；`无` 不能与另外两种行并存。其他自然语言中的“替代”不参与解析。

全部替代关系归一化为有向边 `旧 ADR -> 新 ADR`：

- 在 ADR-A 中写 `replaces ADR-B` 生成边 `ADR-B -> ADR-A`。
- 在 ADR-B 中写 `replaced-by ADR-A` 同样生成边 `ADR-B -> ADR-A`。
- 每条边必须由两端互相声明：A 的 `replaces` 包含 B，且 B 的 `replaced-by` 等于 A；单边声明判无效。
- 旧端 B 的治理状态必须为 `已替代`；出现 `replaced-by` 的 ADR 必须为 `已替代`，状态为 `已替代` 的 ADR 也必须有且只有一个 `replaced-by`。
- 新端 A 不得因 `replaces` 自动改变治理状态；若 A 同时被后续决定替代，则它可以同时具有 `replaces` 和 `replaced-by`，但其状态必须为 `已替代`。
- 所有目标必须存在，不得自引用；按归一化边检查时不得形成环。两端对同一边的重复声明只计一条边，不构成环。

恢复使用的 ADR 索引是从 `DECISIONS.md` 确定性生成的只读投影，不是第二份权威状态。解析器只提取每个 `## ADR-*` 标题、首个治理状态、可选替代关系和正文起始定位，输出：

```text
id | title | governance_state | replaces_ids | replaced_by_id | body_anchor
```

初始恢复上下文只注入该投影，不注入 ADR 原因和决定正文。需要某项决定时，再按 `body_anchor` 读取该章节。解析失败或出现重复 ID、缺失状态、无效替代目标时必须报告，不能退化为把完整 `DECISIONS.md` 静默注入上下文。

## 4. 状态重建流程

```text
先读取 PLAN，恢复紧凑快照、阶段和引用的决定编号
→ 若存在 DECISIONS，先用确定性解析器生成并读取 ADR 元数据投影
→ 将全部已确认且未废弃 ADR 与 PLAN 引用交叉核对；漏引、误引或引用已替代决定时报告
→ 只读取当前任务需要的 ADR 正文
→ 只有 PLAN 引用产品范围/需求或当前任务需要产品基线时，才读取 PRD 相关章节
→ 检查执行状态、治理状态、证据状态和更新时间
→ 若存在预登记审计 ID，标记为生成中或中断残留，不将其作为有效结论
→ 只有需要依赖候选或确认结论时，才读取最新审计清单并重新校验文件哈希；否则标记为尚未复验
→ 提取目标、阶段、有效决定、下一步和风险
→ 检查文件之间的冲突或缺失
→ 输出恢复摘要
→ 对阻塞性冲突请求用户确认
→ 再继续工作
```

恢复摘要只报告实际存在且影响当前工作的类别：

- 已完成。
- 正在执行。
- 待确认。
- 已废弃或已替代。
- 文件缺失或无法判断。
- 已确认且当前有效的决定。
- 证据成熟度和审计状态仅在当前任务依赖这些结论时报告。

## 5. 冲突规则

- PRD 管理产品目标、范围和验收标准。
- DECISIONS 管理已确认的技术和流程决定。
- PLAN 管理当前执行状态。
- ADR 不得静默改变 PRD；如决定影响产品目标或范围，必须同步更新 PRD 并重新确认。
- PLAN 不得改变 PRD 或 ADR，只能引用。
- 任务简报是当前状态的派生视图，不是独立权威来源。
- 只有审计清单中的全部文件哈希仍匹配时，才能恢复“待确认候选”或基于该审计的确认结论。
- 架构审计清单还必须包含 `docs/audits/README.md` 规定的全部静态必需文件；即使已列文件哈希匹配，缺少必需路径也判定无效。动态 `PLAN.md` 只提供 audit ID，不参与静态哈希。
- 恢复待确认候选时验证候选审计完整哈希；恢复已确认基线时，`PLAN.md` 的最新审计必须指向 `CONFIRM-*`，并验证确认凭证当前完整哈希、父候选审计、规范化哈希和治理状态差异。

## 6. MVP 验收

使用一个全新线程，只提供项目目录，不提供原对话：

1. 正确恢复当前目标、阶段、下一步和已确认决定。
2. 不把建议或废弃决定当成当前决定。
3. 发现至少一个人为设置的冲突并报告。
4. 不在阻塞性冲突未解决时继续实施。
5. 关键字段正确率必须为 100%；非关键说明允许措辞差异。

普通恢复关键字段固定为：

- 当前有效目标。
- 当前关键约束。
- 已完成。
- 下一步。
- 阻塞项。

条件关键字段只在对应文件或状态存在时评分：

- 当前阶段和执行状态。
- 已确认且未废弃的决定编号，以及 PLAN 漏引、误引或已替代引用。
- 当前有效正式实现契约及版本历史。
- 当前风险和需要用户决定的事项。
- 最新审计 ID 及其哈希有效性。
- 预登记审计 ID；存在时必须明确报告尚未生效。

每个关键字段按“与夹具完全一致、明确报告缺失或冲突、错误”三类判定。前两类在符合夹具预期时计为正确；不得通过猜测补齐缺失值。正确率按 `正确关键字段数 / 应评关键字段总数` 计算。
