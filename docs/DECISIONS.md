# 决策记录

> 文档阶段：架构  
> 默认日期：2026-06-10  
> 默认适用范围：本项目 MVP 架构与实施  
> 默认证据状态：独立复核  
> 说明：ADR 一致性与确认范围已静态复核

每条 ADR 的治理状态只使用：`建议确认`、`已确认`、`已废弃`、`已替代`。证据状态独立使用：`未检查`、`已自查`、`独立复核`、`已验证`。未单列日期、适用范围或证据状态时继承文档级默认值。

供状态恢复使用的 ADR 索引不单独持久化。它必须按 `STATE_CONTRACT.md` 从每个 `## ADR-*` 标题、首个治理状态、可选替代关系和正文定位确定性生成，避免复制治理状态形成第二权威来源。

替代关系仅识别严格行 `- 替代关系：无`、`- 替代关系：replaces ADR-001, ADR-002` 或 `- 替代关系：replaced-by ADR-003`。同一 ADR 可各有一行 `replaces` 和 `replaced-by`，`无` 不得与其他关系并存。所有关系按 `旧 ADR -> 新 ADR` 归一化，两端必须互相声明；目标存在性、自引用、环和 `已替代` 状态的一致性按 `STATE_CONTRACT.md` 校验。

## ADR-001 产品形态分层

- 治理状态：已确认
- 决定：采用可选 `AGENTS.md`、核心技能、按需参考模块、项目文档、评估体系和插件分发的分层结构。
- 原因：同时满足默认行为、上下文效率、项目持久化、个人使用和后续分享。
- 排除：
  - 单一大型技能：上下文和维护成本过高。
  - 多个平级技能作为首版入口：用户负担和编排冲突较高。

## ADR-002 核心技能数量

- 治理状态：已确认
- 决定：MVP 使用一个核心编排技能。
- 原因：为用户提供单一入口，并统一任务分级、确认、研究和验证。
- 演进：专项能力只有在评估证明独立触发更可靠时才拆分为独立技能。

## ADR-003 详细知识的存放方式

- 治理状态：已确认
- 决定：核心流程写入 `SKILL.md`，详细方法写入一层 `references/`。
- 原因：遵循渐进式加载，降低上下文占用和核心规则膨胀。

## ADR-004 项目状态

- 治理状态：已确认
- 决定：快速任务只使用对话状态；MVP 完整模式按 `docs/STATE_CONTRACT.md` 使用最小项目文档。
- 原因：避免简单任务负担，同时为复杂项目提供可恢复、可审查的基线。
- 限制：插件升级不得覆盖项目文件。
- 长期规则：`AGENTS.md`、技能和全局指导只能先生成候选差异，经用户明确批准后修改。

## ADR-005 多 Agent 默认策略

- 治理状态：已确认
- 决定：默认单 Agent；所有复杂项目必须由核心技能主动进行多 Agent 适用性判断。判断有明显价值时，必须向用户说明角色、边界、收益、成本、风险和单 Agent 替代方案，由用户决定是否启用。
- 原因：符合 Codex 当前明确授权要求，并控制 Token、冲突和协调成本。
- 未授权行为：用户未明确同意时，继续采用单 Agent，不得自行启动子 Agent。
- 授权作用域：默认仅当前阶段有效，记录角色、数量、读写范围和终止条件；扩大范围时重新询问。
- 授权状态：`proposed`、`accepted`、`declined`、`expired`；只有用户明确同意才进入 `accepted`，其他回复继续按单 Agent 推进。
- 限制：MVP 不捆绑自定义 Agent。

## ADR-006 联网研究策略

- 治理状态：已确认
- 决定：专业、时效敏感或存在明显知识缺口的问题默认主动联网；涉及敏感内容时先进行隐私和数据流评估。
- 原因：专业性和时效性需要当前资料，安全性要求最小必要披露。

## ADR-007 MVP 扩展边界

- 治理状态：已确认
- 决定：MVP 不包含 Hooks、自建 MCP、应用连接、自定义 Agent 和周期性自动化。
- 原因：这些能力不是验证核心价值的必要条件，会增加安装、安全和维护成本。
- 演进：只有真实样例显示稳定重复需求时再加入。
- MVP 收缩：插件封装、完整恢复、完整知识治理和复杂配置分层延后到核心技能通过基础评估之后。

## ADR-008 评估优先

- 治理状态：已确认
- 决定：技能封装为插件前，必须通过固定样例和真实任务评估。
- 原因：防止只依据文档完整度判断技能有效，并减少插件化后返工。
- 最低要求：与无技能基线比较；关键触发、确认和安全样例重复运行；记录模型、工具、耗时、干预和主要质量指标。

## ADR-009 开发与分发顺序

- 治理状态：已确认
- 决定：先作为当前仓库的本地技能迭代，再封装为插件并通过个人 Marketplace 安装测试。
- 原因：符合官方“个人或单仓库工作流先用本地技能，稳定后再插件化”的建议。

## ADR-010 默认体验

- 治理状态：已确认
- 决定：
  - 资源：平衡模式。
  - 流程：自动分级。
  - Agent：默认单 Agent；复杂项目主动评估并由用户决定。
  - 联网：需要时主动使用。
  - 输出：结论优先。
  - 文档：完整模式默认创建最小状态文件，其他文档按需增加。
- 原因：在速度、质量、用户负担和可控性之间提供稳健起点。

## ADR-011 提示词产品定位

- 治理状态：已确认
- 决定：不以“提示词大全”或单一万能提示词作为产品核心。用户以自然语言输入，由技能动态生成任务简报和阶段任务书。
- 原因：降低用户学习和选择成本，并根据任务风险、项目状态和工具能力动态调整。

## ADR-012 提示词分层

- 治理状态：已确认
- 决定：
  - 长期规则进入 `AGENTS.md`。
  - 触发边界进入技能 `description`。
  - 稳定决策流程进入核心 `SKILL.md`。
  - 专项协议进入 `references/`。
  - 当前目标进入运行时任务简报。
  - 研究、实施、审查和子 Agent 工作进入阶段任务书。
- 原因：减少冲突和重复，支持渐进式加载与独立评估。

## ADR-013 提示词快捷入口

- 治理状态：已确认
- 决定：MVP 提供自然语言覆盖指令和显式技能调用，不依赖自定义 Prompt 命令。
- 原因：首要环境是 Codex 桌面应用；自定义 Prompt 命令不是核心跨表面能力，且会重新增加用户选择模板的负担。

## ADR-014 提示词优化原则

- 治理状态：已确认
- 决定：从最小可用提示开始，通过正例、边界例、负例和回归样例迭代；不要求输出思维链，不用泛化专家角色代替完成标准。
- 原因：当前模型更适合直接、明确、无冲突的指令，提示词改进应由评估证据驱动。

## ADR-015 执行与评审提示分离

- 治理状态：已确认
- 决定：独立评审使用与执行流程不同的任务书，不传递原实现者的预期结论或辩护。
- 原因：降低同一提示逻辑自我证明正确的风险。

## ADR-016 研究停止条件

- 治理状态：已确认
- 决定：每次专业研究在开始前定义问题、来源优先级、投入预算和停止条件；信息价值趋近于零时停止扩展并报告不确定性。
- 原因：避免“更专业”被错误实现为无上限搜索。

## ADR-017 最小状态重建

- 治理状态：已确认
- 决定：MVP 先从紧凑 `PLAN.md` 重建最小状态；仅在文件存在且当前任务需要时，通过元数据投影读取 `DECISIONS.md` 的相关 ADR，并按引用读取 `PRD.md` 局部章节。完整线程历史恢复延后。
- 原因：先用可定义、可盲测的数据契约验证跨线程连续性。

## ADR-018 联网预检

- 治理状态：已确认
- 决定：公开信息可主动联网；内部信息必须先脱敏；受限数据默认不得进入外部查询。内部信息授权记录项目或任务、数据级别、来源类别、查询类型、允许字段、有效阶段和失效条件；作用域变化时重新确认。
- 原因：数据一旦进入查询就可能已经泄露，不能只在搜索结果返回后评估风险。

## ADR-019 文档职责权威

- 治理状态：已确认
- 决定：PRD 管产品目标和范围，ADR 管技术与流程决定，架构管实现边界，PLAN 管执行状态，任务简报仅为派生视图。ADR 改变 PRD 时必须同步更新并重新确认。
- 原因：单一线性优先级可能让技术决定错误覆盖产品需求。

## ADR-020 评估判定

- 治理状态：已确认
- 决定：MVP 的样例、评分、硬失败、运行次数和通过阈值统一由 `docs/EVALUATION.md` 定义。
- 原因：避免使用“实质改善”等不可操作措辞，也避免执行提示自行证明正确。

## ADR-021 版本控制前置

- 治理状态：已确认
- 决定：实施开始前初始化 Git；若用户不希望使用 Git，则建立带时间戳的只读快照作为最低回滚机制。
- 原因：当前目录不是 Git 仓库，而架构要求变更可追踪和回退。

## ADR-022 审计闭环与状态声明

- 治理状态：已确认
- 决定：高影响产物分别记录治理状态和证据状态。P0/P1 修复后，受影响范围内的旧审计结论失效；必须完成受影响项回归、跨文档一致性检查和反方挑战，才能重新声明为待确认候选。审计记录绑定文件清单、版本或哈希、时间和审查者。架构静态门与技能实现后的行为评估分开。
- 原因：用户审批和证据成熟度并非同一条状态轴；完成一次审查也不代表已穷尽问题。若没有版本绑定和修复后回归，Codex 容易把阶段性检查误报为充分保证。

## ADR-023 受控自主改进

- 治理状态：已确认
- 日期：2026-06-11
- 证据状态：独立复核
- 决定：技能在实际运行中主动采集脱敏改进信号、形成隔离候选并运行回归；长期规则和生效技能只有在用户明确批准后才能更新。开发阶段记录保存在技能源码仓库；插件阶段按项目显式启用，不读取 transcript，只搬运 schema 校验通过的脱敏观察信封到 `PLUGIN_DATA` 项目分区。Hook 定义绑定插件版本、权限声明和脚本摘要。周期性写入自动化必须在独立 Worktree 中使用 `workspace-write`，且唯一可写根为该 Worktree；不能验证隔离时仅生成只读报告。低影响信号批量整理，高风险缺陷立即报告，退化版本和数据迁移必须可回滚。
- 原因：真实使用是改进技能的最佳来源，但直接自我修改容易放大偶然偏好、错误经验和安全缺陷。候选与生效版本分离可以兼顾主动性与控制权。

## ADR-024 经验检索渐进演进

- 治理状态：已确认
- 日期：2026-06-11
- 证据状态：独立复核
- 决定：MVP 以结构化、可版本化的记录作为经验权威来源，先使用字段过滤和全文检索。完成 50 个真实任务、积累 200 条有效经验、发生至少 3 次可复现漏召回、现有标注基准 `Recall@5` 低于 0.70 或用户明确要求时，触发结构化、全文和混合向量检索对比评估。检索必须先执行硬作用域过滤；只有通过定义的召回、精度、零越界、延迟和可重建门，才形成向量召回采用候选。向量索引始终可从结构化记录重建。
- 原因：当前核心问题是经验质量、作用域、审批和回归，而不是大规模语义召回。渐进式引入避免在没有数据和基准时增加嵌入、索引、阈值、隐私和维护复杂度，也保留规模增长后的升级路径。

## ADR-025 结果与实现路径双重验收

- 治理状态：已确认
- 日期：2026-06-11
- 证据状态：独立复核
- 决定：正式实现契约采用 `any-of` 触发表。改变生产入口、状态权威或持久化、架构边界接口或所有权、核心能力或替代结果路径，用户明确要求原路径证明，或变更高影响时，必须在首次相关写入前冻结带版本、来源和时点的契约，并分别通过结果门和实现门。文件数、模块数和保持接口与数据流不变的跨文件修改本身不触发契约；只有全部触发项明确为否时才使用简短内部路径约束。冻结后的修改保留差异，不得事后覆盖；候选契约不能循环证明自身正确。重要功能至少需要一条正式产品入口、非 mock 核心模块和真实状态或等价副作用组成的证据。
- 原因：单纯黑盒验收无法发现影子实现、硬编码结果和绕过既有产品功能的问题。用户不需要理解全部代码，但需要能够判断实现是否沿着认可的产品路径完成。

## ADR-026 运行时薄核心与机制退出

- 治理状态：已确认
- 日期：2026-06-11
- 证据状态：独立复核
- 决定：完整治理体系保留在架构、参考资料、评估和审计中。明确低风险的一步任务不触发完整技能正文；第三方下载、高风险一步操作和其他任务只运行薄核心及命中硬触发或当前目标需要的最小参考章节。上下文预算按实际字符数和章节数计算，协议文件数量仅作辅助指标；供应链安全、授权、实现门、恢复和时效研究不得为了预算被省略。普通任务不自动读取完整治理文件。只有持续生效且具有明显运行成本的机制进入账本，并按 `invariant/safety/quality/convenience` 的可判定标准分类；多类命中取最严格类别，安全不变量只能在先建立等价覆盖、硬失败回归和用户确认后替换或降级，不能通过先降级再删除绕过。
- 原因：可靠性机制只有在净收益为正时才有价值。把全部治理规则放入每次运行会造成上下文污染、流程僵化、速度下降和确认过多，也会削弱真正重要规则的显著性。
