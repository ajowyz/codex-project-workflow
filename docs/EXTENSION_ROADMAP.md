# 后续扩展路线

## 当前基线

当前产品已经达到“个人可用、可复验、可继续扩展”的状态：

- 技能规则已激活。
- 插件源包已创建。
- Codex App 启用后，新线程安装 smoke 已通过。
- 已有只读复验脚本。
- 当前仍是纯技能插件，不包含 Hook、MCP、app connector 或自定义 Agent manifest。

## 近期优先级

### P1 使用文档和模板打磨

目标：让日常使用者不需要理解内部代码，也能正确使用插件。

当前进展：

- 已补充长对话转新线程交接模板。
- 已补充多 Agent 判断模板的 `proposed` 状态、替代方案、后果和未授权可继续工作。
- 已补充专业方案模板的推荐优先、证据版本、适用边界、反证和下一步验证。

动作：

- 继续把真实项目中的好提示词沉淀为模板。
- 后续优先补风险等级和确认边界标注，而不是继续扩写长模板。

### P1 更新和复验流程稳定

目标：每次插件更新后都能快速确认没有误走项目 `.agents` 副本。

当前进展：

- `scripts/verify_plugin_install_smoke.py` 已输出明确的 PASS/FAIL、路径、fallback guard 和 metrics。
- 已新增 smoke 脚本 focused tests，覆盖成功输出、失败提示、混合路径拒绝和临时目录执行。

动作：

- 保持 `scripts/verify_plugin_install_smoke.py` 可运行。
- 每次更新 cachebuster 后都跑安装 smoke。
- 记录新 cache 版本和 metrics。

### P1 真实项目演练

目标：验证插件在真实工作中的行为，不只验证安装。

建议演练：

- 已完成：低风险代码修改，smoke 脚本 focused tests。
- 已完成：需要专业方案比较的模板演练。
- 已完成：需要判断是否启用多 Agent 的模板演练。
- 已完成：实现路径证明模板演练。
- 已完成：新项目需求访谈模板演练。

进入 P2 前的真实项目演练门已满足；首轮安装/更新自动化和真实本地更新闭环已经完成，后续仍不得自动修改已安装 cache、marketplace 或其他敏感配置。

## 中期扩展

### P2 改进候选机制

目标：让插件能从实际使用中学习，但不静默自改。

做法：

- 记录失败、人工干预、重复模式。
- 把值得沉淀的经验做成候选改进。
- 绑定范围、证据、回归和失效条件。
- 通过后再激活。

### P2 安装/更新自动化

目标：减少手工复制和 cachebuster 操作。

当前进展：

- 已新增默认干跑的插件源包更新准备脚本，用于验证 source、target、required files 和 safety 输出。
- 脚本只有显式 `--apply` 才复制仓库插件源包到个人插件源目录；不修改 marketplace 或已安装 plugin cache。
- 脚本已支持 marketplace 预检、target 残留文件防护，以及显式 `--apply --apply-cachebuster` 更新个人插件源 manifest 版本。
- 已新增 `docs/IMPROVEMENT_CANDIDATES.md`，用于把本地使用中发现的问题先沉淀为可审查候选，再进入更新流程。
- DOGFOOD-16 已完成一次真实本地更新闭环：source prep、Codex App re-enable、install smoke、fresh-thread pickup smoke、记录新 cache 版本 `0.1.0+codex.20260703113254`。

可选方向：

- 后续源改动重复同一闭环：候选记录、source prep、Codex App re-enable、install smoke、fresh-thread pickup smoke、记录新 cache 版本。
- 继续保持 marketplace、已安装 cache 和其他敏感配置不被脚本静默修改。
- 下一类值得校准的是 P2 多 Agent 使用判断，或在真实项目中继续收集候选改进信号。

### P2 多 Agent 使用校准

目标：让 Codex 更准确判断什么时候需要多 Agent。

观察点：

- 多 Agent 是否减少遗漏。
- 是否带来过多沟通成本。
- 是否出现职责重叠。
- 是否能明确 fallback。

## 何时考虑向量数据库

当前不需要默认引入向量数据库。

开始专项评估的信号：

- 经验和案例数量明显增多，普通文档难以维护。
- 需要跨项目召回相似任务、失败模式和方案比较。
- 关键词或结构化索引无法稳定找到相关经验。
- 需要基于语义相似度做推荐，而不是只按标题、标签、路径检索。
- 已有明确的隐私、清理、重建、导出和回滚要求。

即使引入，也建议先保持本地优先：结构化记录为主，向量索引用于召回增强，不能替代可审计的源记录。

## 长期方向

### P3 Hook

只有在需要自动提醒、自动拦截或自动记录时才考虑 Hook。

必须先解决：

- 项目显式启用。
- 可查看、可停用、可删除。
- 敏感信息过滤。
- 权限不足时安全降级。

### P3 MCP 或外部服务

只有当本地脚本和文档不足以完成任务时才考虑。

适合场景：

- 需要统一访问外部知识库。
- 需要长期运行的服务。
- 需要跨项目共享状态。

### P3 自定义 Agent manifest

只有当插件需要在 Codex UI 中暴露更明确的专用 Agent 行为时才考虑。

当前默认保持纯技能插件，减少束缚和复杂度。

## 不建议做的事

- 不要把所有项目经验直接塞进技能规则。
- 不要默认联网、默认多 Agent、默认完整治理流程。
- 不要因为模型上下文变大就取消安全和确认门。
- 不要用向量数据库替代清晰的结构化记录。
- 不要让插件静默修改自己。
