## 历史中期扩展（冻结）

### 历史 P2：改进候选机制

目标：让插件能从实际使用中学习，但不静默自改。

做法：

- 记录失败、人工干预、重复模式。
- 把值得沉淀的经验做成候选改进。
- 绑定范围、证据、回归和失效条件。
- 通过后再激活。

### 历史 P2：安装/更新自动化

目标：减少手工复制和 cachebuster 操作。

当前进展：

- 已新增默认干跑的插件源包更新准备脚本，用于验证 source、target、required files 和 safety 输出。
- 脚本只有显式 `--apply` 才复制仓库插件源包到个人插件源目录；不修改 marketplace 或已安装 plugin cache。
- 脚本已支持 marketplace 预检、target 残留文件防护，以及显式 `--apply --apply-cachebuster` 更新个人插件源 manifest 版本。
- 已新增 `docs/IMPROVEMENT_CANDIDATES.md`，用于把本地使用中发现的问题先沉淀为可审查候选，再进入更新流程。
- DOGFOOD-16 已完成一次真实本地更新闭环：source prep、Codex App re-enable、install smoke、fresh-thread pickup smoke、记录新 cache 版本 `0.1.0+codex.20260703113254`。

可选方向：

- 后续源改动重复同一闭环：候选记录、source prep、Codex App re-enable、install smoke、fresh-thread pickup smoke、记录新 cache 版本。
- 继续保持 marketplace、已安装 cache 和其他敏感配置不被脚本静默修改。
- 下一类值得校准的是 P2 多 Agent 使用判断，或在真实项目中继续收集候选改进信号。

### 历史 P2：多 Agent 使用校准

目标：让 Codex 更准确判断什么时候需要多 Agent。

观察点：

- 多 Agent 是否减少遗漏。
- 是否带来过多沟通成本。
- 是否出现职责重叠。
- 是否能明确 fallback。

## 历史评估项：何时考虑向量数据库

当前不需要默认引入向量数据库。

开始专项评估的信号：

- 经验和案例数量明显增多，普通文档难以维护。
- 需要跨项目召回相似任务、失败模式和方案比较。
- 关键词或结构化索引无法稳定找到相关经验。
- 需要基于语义相似度做推荐，而不是只按标题、标签、路径检索。
- 已有明确的隐私、清理、重建、导出和回滚要求。

即使引入，也建议先保持本地优先：结构化记录为主，向量索引用于召回增强，不能替代可审计的源记录。

## 历史长期方向（冻结且未授权）

### 历史 P3 候选：Hook

只有在需要自动提醒、自动拦截或自动记录时才考虑 Hook。

必须先解决：

- 项目显式启用。
- 可查看、可停用、可删除。
- 敏感信息过滤。
- 权限不足时安全降级。

### 历史 P3 候选：MCP 或外部服务

只有当本地脚本和文档不足以完成任务时才考虑。

适合场景：

- 需要统一访问外部知识库。
- 需要长期运行的服务。
- 需要跨项目共享状态。

### 历史 P3 候选：自定义 Agent manifest

只有当插件需要在 Codex UI 中暴露更明确的专用 Agent 行为时才考虑。

当前默认保持纯技能插件，减少束缚和复杂度。

## 历史不建议事项（继续有效）

- 不要把所有项目经验直接塞进技能规则。
- 不要默认联网、默认多 Agent、默认完整治理流程。
- 不要因为模型上下文变大就取消安全和确认门。
- 不要用向量数据库替代清晰的结构化记录。
- 不要让插件静默修改自己。
