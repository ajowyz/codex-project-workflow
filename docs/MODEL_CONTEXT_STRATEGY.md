# 模型与上下文策略

> 状态：草案，可用于后续校准  
> 日期：2026-06-22  
> 适用范围：模型升级、上下文窗口变化、推理能力变化、成本变化后的工作流调整  
> 规则源关系：本文不覆盖 `SKILL.md`、安全边界、确认门或候选激活机制

## 1. 核心原则

模型上下文变大，只改变可用上限，不改变默认治理策略。

保持不变：

- 低风险任务仍走轻量路径。
- 普通任务仍不默认加载 PRD、架构、ADR、评估全集。
- 安全、授权、联网、安装、外部执行、敏感数据和破坏性操作不因模型变强而降级。
- 实现类任务仍分别验证结果和实现路径。
- 长期规则仍通过候选、预检、回归和激活门生效。
- 多 Agent 仍按任务价值判断，而不是按模型上下文大小机械启停。

可以变化：

- 标准模式和完整模式允许的参考资料上限。
- 状态恢复时可读取的历史摘要深度。
- 专业研究中的来源对比数量。
- 方案比较和反例搜索的深度。
- 独立复核、长链审计和批量经验整理的可行性。

## 2. 能力档位

| 档位 | 适用条件 | 默认策略 |
| --- | --- | --- |
| S 小上下文 | 上下文窗口、成本或工具稳定性受限 | 严格薄核心；只读必要协议切片；更多依赖摘要和索引 |
| M 中上下文 | 上下文明显增加，但成本或漂移风险仍显著 | 标准任务可多读少量相邻证据；完整任务仍按阶段加载 |
| L 大上下文 | 长上下文稳定、成本可接受、回归表现不退化 | 完整模式可加载更多证据、历史摘要和方案对比 |
| XL 超大上下文 | 长链审计、跨项目恢复和批量经验整理稳定通过 | 允许更深恢复和批量分析，但仍保留来源索引和确认门 |

档位不是模型名称的同义词。每次模型、推理强度、工具能力、价格或平台行为变化后，都需要重新校准。

## 3. 策略配置字段

后续可以为每个可用模型维护一条策略记录：

```yaml
model_profile:
  id: local-profile-name
  model_name: unknown
  updated_at: YYYY-MM-DD
  context_tier: S | M | L | XL
  reasoning_tier: low | medium | high | unknown
  tool_reliability: low | medium | high | unknown
  cost_tier: low | medium | high | unknown
  default_mode: standard
  limits:
    quick:
      skill_body_chars: 1500
      reference_codepoints: 0
      reference_h2_sections: 0
    standard:
      skill_body_chars: 1500
      reference_codepoints: 2500
      reference_h2_sections: 2
    full:
      skill_body_chars: 1500
      reference_codepoints: 6000
      reference_h2_sections: 4
  hard_invariants:
    safety_and_authorization: unchanged
    implementation_path_verification: unchanged
    candidate_activation: unchanged
  verified_by:
    - evaluation_or_exercise_id
```

默认值应保守。只有校准证明质量不退化、成本可接受、人工干预不增加，才提高档位或放宽上限。

## 4. 升级校准流程

模型或上下文能力变化后，按以下流程处理：

1. 记录变化类型：模型、推理强度、上下文窗口、工具能力、成本、平台行为。
2. 不立即修改 active skill。
3. 选择一组最小对照任务：
   - 快速任务：验证不被治理拖慢。
   - 标准任务：验证按需读取和验证路径。
   - 完整任务：验证长上下文不会导致过度读取或跑偏。
   - E32/E35 类任务：验证硬触发不会被预算吞掉。
4. 对比旧策略和新策略：
   - 成功率。
   - 硬失败数量。
   - 人工干预次数。
   - 上下文码点数和章节数。
   - 首次有效输出时间。
   - 是否出现输出正确但路径错误。
5. 只在结果改善或至少不退化时，生成策略候选。
6. 候选通过预检和定向回归后再激活。

## 5. 不同档位的调整建议

### S 档

- 保持当前 CAND-11 风格的严格预算。
- 优先使用索引、摘要、局部章节和 helper。
- 完整项目恢复先读 `PLAN.md`，再按需读 PRD/DECISIONS。

### M 档

- 标准任务可多读一个相邻文件或一个额外小章节。
- 专业研究可多比较一个候选方案。
- 仍不默认读取完整治理文档。

### L 档

- 完整模式可加载更多历史摘要和证据文件。
- 可把方案比较、反方审查和实现路径证明做得更完整。
- 仍要求输出中说明使用了哪些证据，不能把“读得多”当作证明。

### XL 档

- 可尝试跨项目经验归纳、批量机制账本审查、长链恢复。
- 必须保留来源索引、去重、隐私边界和回滚方案。
- 不允许把个人偏好、项目私有规则或历史对话打包为通用插件默认行为。

## 6. 风险与防护

| 风险 | 表现 | 防护 |
| --- | --- | --- |
| 上下文越大越跑偏 | 读入太多历史和过期决定 | 先恢复最小状态，再按任务加载 |
| 规则膨胀 | 把新能力变成更多默认流程 | 只调整档位参数，不默认扩展核心正文 |
| 成本失控 | 每个任务都做完整审计 | 快速/标准/完整分级保留 |
| 隐私扩大 | 长上下文带入更多敏感材料 | 最小必要读取和脱敏规则不变 |
| 验收错觉 | 输出更完整但路径仍可能错 | 结果验证和实现路径验证分开 |
| 多 Agent 误判 | 认为大上下文能替代独立复核 | 高失败代价仍评估独立复核价值 |

## 7. 本项目当前建议

当前保持 S/M 之间的保守策略：

- active skill 继续保留薄核心。
- 普通任务不因模型升级默认读取更多治理文档。
- 大上下文只先用于完整模式、恢复、研究和审计的可选增强。
- 下一个候选优先解决插件封装路径可移植性，不把模型升级策略写入核心 skill。

后续触发条件：

- 新模型或新 Codex 运行环境明显改变上下文窗口。
- 真实任务显示当前预算造成频繁缺上下文。
- 完整模式中多次因为预算需要人工追加信息。
- 成本下降且回归证明质量不退化。
