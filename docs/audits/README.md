# 架构审计清单格式

审计记录分为：

- 候选审计：`ARCH-YYYYMMDD-NN.md`
- 用户确认凭证：`CONFIRM-YYYYMMDD-NN.md`

候选审计必须包含：

```markdown
# <audit-id>

> 时间：YYYY-MM-DD HH:mm Z
> 审查者：主 Agent | 子 Agent 标识
> 结论：草案保持 | 待确认候选
> 独立性：独立多 Agent | 单 Agent 降级复核

## 文件清单

| 文件 | 完整 SHA-256 | 规范化 SHA-256 |
| --- | --- | --- |
| docs/PRD.md | 完整哈希 | 忽略治理状态行后的规范化哈希 |

## 治理状态清单

| 键 | 当前值 |
| --- | --- |
| document:docs/ARCHITECTURE.md | 待确认候选 |
| fr:FR-17 | 建议确认 |
| adr:ADR-001 | 建议确认 |

## 检查

- 结构与引用
- 需求追踪
- 状态契约
- 独立反方审查

## P0/P1

## P2 与未验证假设

## 工具限制
```

规则：

- 文件清单必须使用完整 SHA-256，不允许截断。
- 架构候选审计必须包含以下全部静态路径，缺少任一项即无效：
  - `docs/PRD.md`
  - `docs/RESEARCH.md`
  - `docs/ARCHITECTURE.md`
  - `docs/PROMPTING.md`
  - `docs/STATE_CONTRACT.md`
  - `docs/EVALUATION.md`
  - `docs/DECISIONS.md`
  - `docs/TESTING.md`
  - `docs/TRACEABILITY.md`
  - `docs/CONFIRMATION_SCOPE.md`
  - `docs/audits/README.md`
- `PLAN.md` 是动态执行状态，不纳入静态哈希；“最新审计”只保存当前有效 audit ID，“预登记审计”只保存正在生成但尚未生效的 ID。
- 在计算候选审计哈希前，待确认的架构文档必须已经处于 `待确认候选`，证据状态必须反映实际复核强度。审计后不得再通过修改状态进入候选。
- 清单内任一静态文件完整哈希变化后，候选审计不再匹配当前文件。
- `docs/PLAN.md` 的“最新审计”字段只能指向已经存在且验证通过的当前有效清单；不得指向预定或尚未生成的文件。
- 为避免幽灵审计和自引用，先保存目标文档的原治理状态，在 `PLAN.md` 的“预登记审计”写入预定 audit ID，并将目标静态文档设为候选状态；再计算全部静态文件哈希并创建同 ID 的审计清单。清单验证通过后，原子地将“最新审计”更新为该 ID，并把“预登记审计”清空。审计文件本身不纳入自己的哈希。
- 候选审计生成失败、校验失败或发现未关闭 P0/P1 时，原子恢复保存的文档治理状态并清空“预登记审计”；“最新审计”保留仍有效的上一清单，若上一清单已因静态文件变化失效则设为 `无`。不得遗留没有有效审计支撑的 `待确认候选` 状态。
- 状态恢复时重新计算清单文件哈希；不匹配时报告失效，不沿用结论。

用户确认后允许确认清单中列明的治理状态发生变化。此时生成确认凭证：

- 引用有效的父候选审计 ID。
- 记录用户确认时间及实际接受的 FR、ADR 和文档。
- 对当前静态文件记录新的完整 SHA-256。
- 同时计算“忽略治理状态行后的规范化 SHA-256”，并要求与父候选审计一致。
- 从父候选审计读取完整治理状态清单，记录当前清单和逐项前后差异。
- 每个状态变化必须与 `docs/CONFIRMATION_SCOPE.md` 中用户实际接受的键和目标状态精确匹配；未列出的变化、漏改或目标状态不符都会使凭证无效。
- 若规范化哈希变化，说明存在实质内容修改，必须重新执行候选审计，不能生成确认凭证。

治理状态字段使用严格语法：

- 文档：只解析文件开头元数据块中的第一个 `> 治理状态：<值>`
- PRD：只解析每个 `### FR-*` 章节中的第一个 `**状态：<值>**`、`**治理状态：<值>**`，以及可选的第一个 `**架构落实状态：<值>**`
- ADR：只解析每个 `## ADR-*` 章节中的第一个 `- 治理状态：<值>`

状态字段不得在同一行附加原因、条件或政策文字；这些内容必须拆到单独字段。`> 文档治理状态：...` 不属于本次可变治理字段，始终参与实质内容哈希。
代码块、模板示例和其他章节中出现的相同文本不作为治理状态，也不得被规范化。

规范化算法固定为：

1. 按 UTF-8 解码文本；若文件以 UTF-8 BOM 开头，将 BOM 作为编码标记消费，不让它进入规范化文本。
2. 将 `CRLF` 和单独的 `CR` 统一为 `LF`。
3. 只对符合严格语法的治理状态行，将 `<值>` 替换为固定占位符 `<GOVERNANCE_STATE>`；不删除前缀、后缀或整行。
4. 保留其余字符和行顺序，并以单个末尾 `LF` 计算 SHA-256。

证据状态、正文、验收标准、决定内容和确认范围不属于可忽略内容。

确认事务：

1. 保留父候选 audit ID 和候选治理状态清单。
2. 只应用用户接受的状态变化。
3. 生成并验证 `CONFIRM-*` 凭证，包括完整哈希、规范化哈希和完整状态差异。
4. 确认事务开始时将目标 `CONFIRM-*` ID 写入“预登记审计”；验证通过后，将“最新审计”更新为该 ID并清空预登记。
5. 任一步失败时恢复候选治理状态和父候选 audit ID，并清空预登记；不得留下半生效确认。
