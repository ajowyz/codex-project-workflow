## 1. 标记说明

需求条目的治理状态：

- **已确认**：用户已经明确表达。
- **建议确认**：Codex 根据需求与官方能力提出，尚待用户确认。
- **待确认**：会影响产品方向，但当前不阻塞 PRD 审阅。
- **已废弃**：不再适用，仅保留历史原因。
- **明确不做**：不属于本项目目标。

证据状态独立记录为：**未检查、已自查、独立复核、已验证**。用户确认和行为验证不是同一含义；PRD 条目标题下的“状态”默认指治理状态。

## 2. 背景和问题定义

### 已确认

用户希望更好、更高效、更严谨地使用 Codex。当前主要问题包括：

- 用户不知道怎样用 Codex 更容易准确理解的语言和结构表达任务。
- Codex 的规划可能不完整，并可能在项目推进中偏离真实目标。
- 用户不清楚哪些节点需要人工确认。
- Codex 往往被动回答，没有主动从专业角度识别问题、查阅资料、比较方案和提出建议。
- Codex 可能只依赖当前上下文或已有记忆，没有主动查阅网络上已有及最新的资料、项目、文献和实践。
- Codex 可能只给出最先想到的方案。例如面对“经验库”需求时只提出 JSON 文件，而没有主动比较结构化文件、关系数据库、全文检索、向量检索和混合方案的适用条件。
- 现有协作容易变成用户不断追问、Codex逐项补充的“挤牙膏”模式。

本项目要建立一套可复用的 Codex 协作工作流，使 Codex 能主动完成需求理解、完整性扫描、专业调研、方案比较、规划、执行、验证、审查和纠偏。

## 3. 项目目标

### 已确认

1. **提高交互效率**：用户可先用自然语言描述想法，系统主动整理为 Codex 易于执行的任务表达。
2. **提高准确性**：区分用户需求、事实、推断、建议和待验证假设，降低误解与跑偏。
3. **提高专业性**：面对具体项目时主动获取相关资料，并给出有依据的专业建议和方案。
4. **提高严谨性**：按任务风险采用合适的规划、确认、验证和审查流程。
5. **降低用户负担**：从“用户问一句、Codex 补一点”转为 Codex 主动扫描问题空间、提出完整判断，只让用户决定真正需要决定的事项。
6. **支持持续扩展**：最终产品形态包括技能、插件和人类可查阅文档；MVP 先验证仓库级技能和文档，稳定后再封装插件。
7. **支持持续学习完善**：根据具体任务主动查阅已有及最新资料、项目、文献和方法论，将经过核实的经验沉淀为可追溯、可审查的改进。

## 4. 非目标

### 明确不做

- 不建立让 Codex“精通所有领域”的全领域知识库。
- 不把向量库或任何单一技术当作所有知识、经验或 AI 项目的默认答案。
- 不保证 Codex 的结论永远正确。
- 不替代用户在业务、安全、法律、工程和商业上的最终责任。
- 不要求所有简单任务都执行完整 PRD、架构和里程碑流程。
- 不在没有明确授权的情况下自动启动子 Agent。
- 不把“自主学习”解释为模型自行训练、秘密保存用户数据或未经审查永久修改自身行为。
- 不因网络上出现新方案就自动改变已经确认的项目目标、需求或关键架构。

### 建议确认

- 首版不自建搜索引擎、通用知识库或 MCP 服务。
- 首版不使用复杂 Hooks 强制拦截所有行为，先验证指导型工作流是否有效。

## 5. 目标用户

### 已确认

- 首要用户：项目发起人本人。
- 后续用户：希望以较低学习成本改善 Codex 协作质量的其他个人或团队成员。
- 首要使用环境：Codex 桌面应用。

## 6. 核心使用场景

### 已确认

1. 从模糊想法启动新项目。
2. 在已有项目中实现功能、修复问题或继续迭代。
3. 研究技术、工具、产品或其他专业问题。
4. 比较方案并作出技术或项目决策。
5. 审查代码、设计、文档和项目方向。
6. 项目出现遗漏或跑偏后重新建立可信基线。
7. 将有效协作经验沉淀为以后可复用的规则。

## 7. 产品原则

### 建议确认

1. **先主动判断，再提问**：Codex 应先说明当前理解、问题空间、初步建议和真正的阻塞项。
2. **问题按影响排序**：只向用户询问会显著影响目标、范围、风险或成本的事项。
3. **非阻塞问题不打断**：可安全推断的内容应标注假设后继续推进。
4. **流程与风险匹配**：避免简单任务流程过重，也避免复杂任务未经规划直接实施。
5. **专业来自证据**：专业性由资料、方案比较、约束分析和验证体现，而不是自信措辞。
6. **主 Agent 对结果负责**：即使使用多 Agent，也由主 Agent 统一目标、整合证据、处理冲突并报告结论。
7. **纠正必须沉淀**：用户指出的偏差应进入需求、规则、决策记录或回归测试。
8. **持续研究而非静态知识**：涉及可能变化或存在专业方案空间的问题，应主动检查已有和最新资料及相关方法论。
9. **学习结果可追溯**：新知识只有在来源、适用性和验证状态明确后，才可沉淀为长期规则。
10. **方法服务于问题**：不得机械套用知名框架或“最佳实践”，应根据当前项目进行选择、裁剪或组合。
11. **先理解现状再提出改变**：已有项目应先检查代码、文档、环境和历史决定，避免脱离现实重新设计。
12. **主动寻找反证**：重要结论不仅要寻找支持证据，还要寻找反例、失败案例和不适用条件。
13. **用户认知负担最小化**：输出应先给结论、影响和推荐，再提供必要细节，不把内部分析负担转嫁给用户。
14. **能力不足时可降级**：工具、网络或多 Agent 不可用时，应说明影响并采用可行替代流程，而不是假装已经完成。
15. **治理不等于运行负担**：完整规则、审计和评估可以保留，但普通任务只加载完成当前工作所需的最小上下文。

## 8. 用户流程

### 建议确认

1. 用户以自然语言描述任务。
2. 技能隐式触发，或由用户显式调用作为兜底。
3. Codex 先执行不展示模板的轻量分流。
4. 快速模式直接执行并验证；标准或完整模式按需生成任务简报。以下字段是可选工作集，不要求每次全部展示：
   - 目标
   - 交付物
   - 背景和约束
   - 非目标
   - 当前事实与假设
   - 成功标准
5. 标准或完整模式只检查当前任务实际相关的风险维度，不逐项朗读固定清单；候选维度包括：
   - 需求和边界
   - 专业资料与方案空间
   - 数据、权限、安全和隐私
   - 单 Agent 或多 Agent
   - 模型、工具和资源投入
   - 工具、环境和外部依赖
   - 上下文、项目状态和历史决定
   - 验证、恢复和长期维护
6. 只有阻塞性问题才请求用户确认，同时给出建议答案和影响。
7. Codex 按确认后的流程调研、规划、实施和验证。
8. 完整模式按最小状态契约维护项目文档。
9. 完成后区分已验证、部分验证、未验证和需用户验收的内容。

## 9. 功能需求

### FR-01 自然语言任务整理

**状态：已确认**

系统应将用户的自然描述整理为结构化任务，而不要求用户先掌握复杂提示词。

验收标准：

- 能输出目标、交付物、约束、非目标、成功标准和待确认事项。
- 不得把 Codex 建议写成用户已经确认的需求。
- 已提供的信息不得反复询问。

### FR-02 主动完整性扫描

**状态：已确认**

系统应主动发现遗漏、歧义、隐含假设、失败方式、替代方案和长期风险，不能等待用户逐项追问。

验收标准：

- 在提问前先展示当前判断和遗漏扫描结果。
- 不以连续的小问题代替系统分析。
- 对非阻塞项给出带标签的合理默认值。

### FR-03 任务风险分级

**状态：已确认**

系统应先判断任务风险，再选择工作模式。风险等级是系统判断，工作模式是执行深度，用户可以覆盖工作模式。

风险等级：

| 级别 | 判定 |
| --- | --- | --- |
| 低 | 范围明确、影响局部、容易回退、无需专业决策 |
| 中 | 多步骤、存在有限不确定性或影响多个组件，但可分阶段验证 |
| 高 | 目标或需求模糊、跨领域、影响数据/权限/架构、难回退、长期项目或失败代价高 |

工作模式：

| 模式 | 默认适用 | 流程 |
| --- | --- | --- |
| 快速 | 低风险 | 内部轻量分流后直接执行并验证，不展示任务简报、不加载非安全类参考模块 |
| 标准 | 中风险 | 简短任务简报、必要调研、计划、执行和验证 |
| 完整 | 高风险或复杂项目 | 需求、调研、架构、计划、分阶段实施、复核和用户验收 |

验收标准：

- 标准或完整模式说明分级结果及原因；快速模式只做内部路由，不增加可见流程说明。
- 用户可用“快速模式”“完整模式”等指令覆盖默认分级。
- 简单任务不得被迫经过完整项目流程。
- 工作模式控制流程和输出深度，不覆盖安全执行深度；高风险任务仍必须满足最低规划、备份、权限确认和验证要求。

### FR-04 专业调研

**状态：已确认**

面对具体项目中的专业问题，系统应主动获取与该问题相关的当前资料。

验收标准：

- 优先使用官方文档、标准、论文、厂商资料和可靠一手来源。
- 核对版本、日期和适用范围。
- 区分已核实事实、推断、建议和待验证假设。
- 重要结论提供来源。
- 对无法核实的内容明确说明不确定性。
- 根据任务需要检索已有及最新的官方资料、标准、论文、开源项目、方法论、案例和版本变更。
- 记录关键来源的发布日期或版本、查阅日期与适用范围。
- 搜索现有项目时评估维护状态、许可证、社区活跃度、安全风险、依赖成本和与当前需求的匹配度。
- 查阅论文或文献时区分同行评审论文、预印本、厂商研究、二手解读和未经验证观点。
- 研究开始前定义与任务风险匹配的范围、预算和停止条件。
- 当新增来源不再带来实质信息，或剩余问题必须通过实验、原型或用户决策解决时，应停止搜索并总结不确定性。

### FR-04A 方法论发现与裁剪

**状态：已确认**

系统应主动识别当前任务可能适用的方法论、标准流程、评估框架和行业成熟实践，并根据项目约束进行选择、裁剪或组合。

验收标准：

- 区分通用协作方法、专业领域方法和当前项目的实际工作方法。
- 说明每种候选方法论解决的问题、成立前提、适用范围和局限。
- 对照项目规模、风险、预算、人员、时间和生命周期评估适用性。
- 不得仅因方法知名、流行或被称为“最佳实践”就采用。
- 当多个方法可组合时，说明各自承担的职责及可能冲突。
- 优先通过小范围试验验证尚不确定的方法。
- 将采用、裁剪和不采用的理由写入决策记录。

### FR-05 方案空间与专业建议

**状态：已确认**

系统不得默认采用第一个可行方案，应识别有实质差异的合理候选。

验收标准：

- 比较候选方案的适用条件、成本、限制、风险、维护和验证方式。
- 若只推荐一个方案，应说明其他主要路线为何被排除。
- 推荐必须结合当前项目约束，不能只复述通用优缺点。
- 在“经验库”样例中，应根据规模、查询模式、语义检索需求、更新频率、成本和复杂度比较文件、数据库、全文检索、向量检索及混合方案。

### FR-06 人工确认边界

**状态：已确认**

以下事项默认需要用户确认：

- 改变项目最终目标或核心范围。
- 关键业务规则。
- 影响较大的架构选择。
- 不可逆或难回滚操作。
- 数据删除、迁移、覆盖或外泄风险。
- 权限、安全、隐私、法律或显著费用风险。
- 超出已确认范围的大量新增工作。
- 最终业务验收。

以下事项建议默认自主处理：

- 只读探索和资料整理。
- 已确认范围内的常规实现选择。
- 可回退的局部修改。
- 测试、静态检查和结果汇总。

验收标准：

- 请求确认时说明推荐选项、其他选项及不确认的后果。
- 不因低风险细节频繁打断用户。

### FR-07 单 Agent 与多 Agent 策略

**状态：已确认**

系统应评估任务是否适合单 Agent 或多 Agent。

验收标准：

- 评估任务是否可并行、是否需要不同专业视角、是否需要独立审查，以及额外成本是否值得。
- 简单任务默认单 Agent。
- 满足任一条件即视为需要执行多 Agent 适用性判断：跨两个以上相对独立的专业或模块、可并行的只读任务预计超过两个、需要独立复核、预计跨多个里程碑，或失败代价较高。
- 复杂项目必须在正式规划前完成一次多 Agent 适用性判断。
- 复杂的只读探索、调研、测试或审查可建议多 Agent。
- 判断多 Agent 有明显价值时，Codex 必须主动向用户提出，而不是等待用户询问。
- 提议必须说明建议的 Agent 角色、任务边界、预期收益、额外成本、潜在冲突和不使用多 Agent 的影响。
- 并行写入同一范围时默认谨慎，明确文件或模块所有权。
- 是否启用多 Agent 最终由用户决定；未获得明确同意时继续使用单 Agent。
- 授权必须区分 `proposed`、`accepted`、`declined` 和 `expired`；用户补充需求、继续讨论或未回应不构成同意。
- 主 Agent 汇总冲突观点，并按证据、约束和风险给出判断。

### FR-08 项目文档生成与维护

**状态：已确认**

复杂项目允许自动创建和维护普通项目文档：

- `docs/PRD.md`
- `docs/ARCHITECTURE.md`
- `docs/PLAN.md`
- `docs/DECISIONS.md`
- `docs/TESTING.md`
- 必要的变更与验收记录

`AGENTS.md`、核心技能、用户全局指导和插件默认规则不属于可自动维护的普通项目文档，只能生成候选差异并等待用户批准。

验收标准：

- 只有已确认内容才能作为既定需求或决定。
- 建议和假设必须有状态标记。
- 用户纠正后同步更新受影响的文档和测试。
- MVP 新线程可以按照 `docs/STATE_CONTRACT.md` 重建最小状态；完整历史和自动冲突恢复属于后续能力。

### FR-09 实施、验证与审查

**状态：已确认**

系统应根据任务风险执行适当的实现、测试、真实运行和审查。

验收标准：

- 明确哪些结论由测试、实际运行、静态检查或资料证明。
- 明确哪些内容仍未验证或需要人工验收。
- 高风险或重要版本应执行独立反方审查。
- 测试通过不得被描述为业务目标已经自动成立。

### FR-10 纠偏与经验沉淀

**状态：已确认**

用户纠正、失败案例和重复问题应形成持久化改进。

验收标准：

- 分析纠正影响的需求、架构、代码、测试和文档。
- 将稳定规则整理为候选差异；用户批准后再写入 `AGENTS.md`、技能参考资料或其他长期规则。
- 对可复现错误增加回归样例。
- 避免把一次性偏好无限扩展为全局规则。

### FR-11 自动触发与显式控制

**状态：已确认**

需要工作流的普通多步骤项目任务应能通过技能描述隐式触发，同时提供显式调用和退出方式；明确低风险的一步任务不应加载完整技能正文。

验收标准：

- 对项目规划、开发、研究、方案比较、审查和纠偏类任务进行隐式匹配。
- 不干扰闲聊、简单问答和明确低风险的一步修改或命令。
- 用户可以显式调用技能。
- 用户可以要求本次使用快速模式、完整模式、单 Agent 模式或停用该流程。
- 使用样例验证隐式触发率，并记录未触发和误触发情况。

### FR-12 人类查阅文档

**状态：已确认**

项目应提供一份便于用户查阅的使用文档。

验收标准：

- 说明适用场景、工作模式、确认边界、单/多 Agent 策略和常用表达方式。
- 提供短小的实际示例，而不是只提供抽象原则。
- 文档与技能实际行为一致。

### FR-13 可扩展机制

**状态：已确认**

后续应能够增加新流程、模板、研究规则、工具依赖和 Agent 角色。

验收标准：

- 核心技能保持精简，详细内容按需加载。
- 扩展内容有清晰边界，不需要重写全部核心流程。
- 插件升级不应覆盖用户项目内已经确认的需求和决策。

### FR-14 持续学习与自我完善闭环

**状态：已确认**

系统应在执行具体任务时主动发现知识与方法缺口，查阅网络上已有及最新的资料、项目、文献和方法论，并利用经过验证的发现完善当前方案与后续工作流。

验收标准：

- 研究过程遵循“发现缺口、检索、来源评估、方案比较、验证、提出改进、审查后沉淀”的闭环。
- 对时效敏感的问题必须重新查询，不得只依赖模型记忆。
- 新发现应说明它改变了哪些既有判断，以及为什么适用于当前项目。
- 新方法论应说明是直接采用、裁剪采用、组合采用还是不采用，以及判断依据。
- 互相冲突的资料应展示主要分歧、证据质量和当前判断。
- 未经验证的网络观点不得写成已确认事实。
- 可复用结论应在适当确认后进入项目文档、决策记录、回归样例或技能参考资料。
- 不得未经用户确认改变项目目标、核心范围、关键架构或安全边界。
- 用户可以要求针对指定主题进行一次性深度研究。

### FR-15 可选的周期性资料跟踪

**状态：建议确认**

对于长期项目或快速变化领域，系统可在用户明确启用后，通过 Codex 桌面应用自动化定期检查新资料。

验收标准：

- 启用前明确主题、来源范围、频率、停止条件和通知阈值。
- 只报告与当前项目有实质影响的新内容，避免重复信息。
- 输出新增、变化、失效和存在争议的内容，并附来源。
- 默认只提出更新建议，不自动修改代码、需求或架构。
- 用户可以随时暂停或删除周期性跟踪。

### FR-16 上下文与完整线程恢复

**状态：建议确认**

系统后续应在 MVP 最小状态重建基础上，增强长对话、线程切换和历史信息过期时的完整恢复能力。

验收标准：

- 开始已有项目任务时先检查适用的项目规则、需求、计划、决策和相关代码。
- 区分当前有效信息、历史信息、已废弃决定和未经确认内容。
- 重要假设、决定、风险和未完成事项不只保存在聊天记录中。
- 线程恢复时先生成当前状态摘要，再继续实施。
- 对话内容与项目文件冲突时指出冲突，不默认任何一方正确。
- 不为简单任务创建不必要的持久化文件。

### FR-17 模型、工具与资源策略

**状态：已确认**

系统应根据任务复杂度、速度要求、验证难度和成本选择合适的模型、工具、单/多 Agent 及研究深度。

验收标准：

- 说明明显影响结果的工具或资源选择理由。
- 简单任务优先采用低开销流程。
- 复杂推理、高风险决策和独立审查可以建议更高质量投入。
- 搜索、浏览器、MCP、脚本、测试工具和子 Agent 应按实际价值使用，不为展示能力而使用。
- 在投入继续增加但预期收益很低时停止扩展，并报告剩余不确定性。
- 用户可选择速度优先、平衡或质量优先。

### FR-18 研究完整性与认知偏差控制

**状态：已确认**

系统应降低只寻找支持当前想法的资料、追逐流行方案或被单一来源影响的风险。

验收标准：

- 对重要判断主动搜索反例、失败案例和不支持证据。
- 区分来源的独立性，避免将互相转载的内容误认为交叉验证。
- 识别营销材料、基准选择偏差、幸存者偏差和过时结论。
- 方案比较使用与当前项目一致的评价维度。
- 不以搜索结果数量代替证据质量。
- 对证据不足但必须推进的事项明确记录临时假设和验证计划。

### FR-19 失败、降级与恢复

**状态：已确认**

系统应为网络不可用、资料冲突、工具失败、权限不足、上下文缺失、验证失败和执行中断提供可理解的处理方式。

验收标准：

- 不得把工具未执行或结果不可见描述为已经验证。
- 失败后说明失败位置、已产生的影响、可恢复状态和下一步。
- 能安全回退时优先回退；不能回退时立即停止扩大影响。
- 网络不可用时区分可基于本地资料继续的内容与必须等待核实的内容。
- 多 Agent 或高级工具不可用时可退回单 Agent 流程，并说明质量或速度影响。
- 中断恢复后检查外部状态是否已变化，不盲目重复高风险操作。

### FR-20 效果评估与回归测试

**治理状态：已确认**

**实现状态：MVP 基础版本由 `docs/EVALUATION.md` 定义**

系统自身应通过真实样例和可重复评估证明其改善了 Codex 协作，而不能只凭文档判断有效。

验收标准：

- 建立覆盖快速、标准、完整、已有项目、专业研究和纠偏的评估样例集。
- 至少评估：
  - 隐式触发是否正确
  - 需求复述是否准确
  - 是否出现重复提问
  - 是否主动发现关键遗漏
  - 方案空间是否合理完整
  - 是否存在无来源的重要断言
  - 人工确认时机是否合适
  - 是否发生未授权范围扩大
  - 新线程恢复是否正确
- 每次修改技能后运行相关回归样例。
- 评估失败应保留原始输入、输出、失败原因和修正结果。
- 不得只使用为当前技能量身定制的理想样例，应包含模糊、冲突和信息不足的任务。

### FR-21 规则、知识与版本治理

**状态：已确认**

系统应控制规则和知识的新增、冲突、过期与升级，避免技能随着使用不断膨胀或形成互相矛盾的要求。

验收标准：

- 新规则进入长期配置前说明来源、适用范围和触发条件。
- 一次性用户偏好、项目规则和跨项目通用规则分别保存。
- 规则冲突时按用户当前明确指令、项目规则和通用规则的适用层级处理，并报告重要冲突。
- 时效敏感知识包含查阅日期或版本，并可标记为待复核、已失效或已替代。
- 定期删除重复、无效或已被更高层规则覆盖的内容。
- 插件或技能升级有版本记录、兼容性说明和回滚方法。
- 不直接覆盖用户项目中的需求、决定和本地定制。

### FR-22 交互输出协议

**状态：已确认**

系统应采用稳定但不过度僵化的沟通结构，让用户快速理解当前状态和需要采取的行动。

验收标准：

- 标准或完整任务开始时说明当前理解、任务级别、计划和必要确认项；快速任务默认不展示这些内部路由信息。
- 工作期间提供简短、有信息量的进度更新。
- 请求用户确认时明确推荐、选项、影响和默认行为。
- 完成时优先报告结果、验证证据、未验证内容和剩余风险。
- 不输出与当前决策无关的大段模板化内容。
- 用户可以要求更简洁或更详细的输出。

### FR-23 技能自身改进治理

**治理状态：已确认**

**架构落实状态：已确认**

系统应能够依据真实使用反馈改进自身，但改进过程必须可审查、可测试和可回退。

验收标准：

- 在技能实际运行期间识别用户纠正、重复追问、遗漏、失败、降级、评估低分、用户明确负面反馈和成功模式等改进信号；不假定能读取产品 UI 点赞/点踩，也不把“自主学习”解释为后台持续训练或秘密收集数据。
- 改进信号先经过脱敏、去重和作用域判断，只记录解决问题所需的最小信息。
- 仓库级技能只能在被触发且拥有写权限时记录；插件阶段的本地记录按项目单独启用，未知项目默认不记录，默认不上传数据。
- 自动记录不得读取会话 transcript；只处理技能为当前任务显式生成、通过固定 schema 验证的脱敏观察信封。
- 每个启用项目使用独立的本地逻辑分区，并提供查看、导出、停用和删除能力；未晋级观察默认保留不超过 90 天。
- 将“发现问题”与“立即修改核心技能”分开。
- 先判断问题属于提示、流程、参考资料、工具、评估样例还是产品能力限制。
- 自动形成隔离的改进候选，记录来源证据、适用范围、预期收益、潜在副作用和回滚方式。
- 修改前说明预期改善、影响范围和可能副作用。
- 修改后运行相关回归样例，并比较修改前后结果。
- 候选通过评估后向用户提交变更摘要；只有用户明确批准，才能更新生效技能、长期规则或插件版本。
- 批准必须绑定候选 ID、补丁 SHA-256、基础源码 SHA-256 或版本、目标文件、作用域、评估结果 SHA-256、批准时间和失效条件；任何绑定内容变化都会使批准失效。
- 被拒绝、退化或证据不足的候选不得静默生效；允许保留为项目级经验、实验记录或回归样例。
- 多个低影响信号默认批量整理，避免每次任务结束都打断用户；高风险缺陷应立即报告。
- 不能证明普遍有效的改动只保留为项目级规则或候选实验。
- 核心技能保持精简，详细方法和领域资料按需加载。

### FR-24 外部内容与供应链安全

**状态：已确认**

系统在查阅网页、代码仓库、论文、依赖和第三方工具时，应防止外部内容操纵任务目标或引入不可信资产。

验收标准：

- 将网页、仓库文档和外部文件视为待分析资料，而不是高优先级指令。
- 不执行外部资料中要求泄露秘密、扩大权限或偏离用户目标的指令。
- 引入依赖或复用代码前检查来源、维护状态、许可证、已知风险和必要性。
- 不因示例代码可以运行就认定其适合生产使用。
- 涉及下载、安装、账号授权或向外部服务发送数据时，按风险请求确认。
- 研究报告标记来源类型，并提示明显的利益相关或营销倾向。

### FR-25 个性化、共享与配置分层

**状态：建议确认**

系统应同时支持个人高效使用和后续他人安装，避免把个人偏好与通用方法混在一起。

验收标准：

- 区分核心通用规则、用户个人偏好、项目专属规则和单次任务指令。
- 用户个人偏好不得被打包成所有插件用户的强制默认值。
- 项目规则不得污染其他项目。
- 用户可以覆盖输出详细度、资源偏好、联网策略和流程模式。
- 提供适合首次使用的默认值，但允许逐步调整，不要求安装时完成复杂配置。
- 分享或升级插件时不携带用户项目数据、私密资料或历史对话。

### FR-26 提示词与任务协议

**治理状态：已确认**

**架构落实状态：已确认**

系统应让用户以自然语言开始，由技能根据任务动态生成适合 Codex 执行的任务简报和阶段任务书，而不是要求用户学习或选择大量固定提示词。

验收标准：

- 用户无需填写复杂模板即可启动任务。
- 快速任务至少明确目标、相关上下文、约束和完成标准。
- 复杂任务补充非目标、事实、假设、风险、工作模式和确认项。
- 研究、实施、测试、审查和子 Agent 委派使用有明确范围、交付物、验证及停止条件的阶段任务书。
- 长期规则、技能触发、核心流程、专项协议和当前任务内容分别保存，不集中到一个万能提示词中。
- 提示词指令直接、明确且不存在已知冲突。
- 不以泛化专家角色或“逐步思考”要求代替资料、评价标准和验证。
- 用户可通过简短自然语言覆盖流程、联网、资源和 Agent 模式。
- 触发描述、任务简报和阶段任务书都有独立评估样例。

### FR-27 持久化规则变更确认

**状态：已确认**

系统可以自主更新普通项目工作文档，但不得未经用户明确确认修改会长期影响后续行为的规则。

验收标准：

- `AGENTS.md`、核心技能、用户全局指导和插件默认规则的变更必须先形成候选差异。
- 候选差异说明来源、适用范围、预期收益和潜在副作用。
- 批准凭证必须绑定候选 ID、补丁哈希、基础版本或源码哈希、目标文件、作用域、评估结果哈希和失效条件；应用前重新校验。
- 用户明确批准后才能写入长期规则。
- 未获批准的改进可以保留在当前任务记录或候选清单中，不得静默生效。
- PRD、计划、研究记录等普通项目文档可在已确认范围内同步维护。

### FR-28 审计闭环与结论校准

**状态：已确认**

系统不得因为完成一次检查就过早宣称高影响成果已经可确认或已验证；问题修复后必须重新验证原结论仍然成立。

验收标准：

- 分开记录治理状态“草案、待确认候选、已确认、已废弃”和证据状态“未检查、已自查、独立复核、已验证”。
- P0/P1 问题修复后，修复前的审计结论自动失效。
- 重新声明为待确认候选前，完成受影响项回归、跨文档一致性检查和至少一次反方挑战。
- 状态声明同时列出检查范围、文件版本或哈希、证据、未验证假设、剩余风险和适用边界。
- 用户要求再次检查时，优先分析上次检查的盲区和结论依据，不只是重复同一检查表。
- P0/P1/P2/P3 使用统一严重度定义；若连续两轮复核仍发现新的 P1，保持草案并由用户决定继续审计、缩小确认范围或暂停。
- 架构静态审计与技能实现后的行为评估分开，不能形成“未确认不能实现、未实现不能确认”的循环。

### FR-29 经验记忆与检索演进

**治理状态：已确认**

**架构落实状态：已确认**

系统应以可审计的结构化记录保存有效经验，并根据真实规模和检索效果决定是否引入全文或向量检索，不预设向量数据库为默认答案。

验收标准：

- 首版保存结构化的改进信号、候选、决策、适用范围、证据状态、技能版本和回归结果；生效规则仍以版本化源码和已批准记录为权威来源。
- 未批准候选、已拒绝经验和已失效规则不得作为可直接执行的通用经验。
- 跨项目沉淀前必须脱敏并抽象为通用模式；项目私有内容默认保留在项目作用域。
- 检索结果只是候选上下文，必须再次检查作用域、版本、证据状态和当前任务约束，不能直接覆盖已确认需求或规则。
- 项目、用户、批准状态、失效状态和数据级别必须在全文或向量召回前执行硬过滤；不允许依赖相似度排序修复越界结果。
- 完成 50 个真实任务或积累 200 条有效经验后触发一次检索方案评估；出现至少 3 次可复现的漏召回、当前标注基准 `Recall@5` 低于 0.70 或用户明确要求时可提前评估。
- “真实任务”指技能实际参与且形成最终结果的一次用户任务，不计评估运行、失败重试或同一任务的恢复；“有效经验”指已经脱敏、去重、完成作用域与证据标注的唯一记录。
- 评估至少比较结构化过滤、全文检索和“结构化过滤 + 全文检索 + 向量召回”的混合方案；达到触发条件只要求评估，不代表必须采用向量检索。
- 只有混合方案通过下列零越界、召回、精度、延迟、成本、隐私和可重建门时才引入。
- 检索升级专项必须满足：硬过滤后 Top-5 错误作用域结果为零；混合方案相对最佳非向量基线的 `Recall@5` 至少提高 10 个百分点、`Precision@5` 下降不超过 5 个百分点；本地 P95 检索延迟不超过 1 秒且不超过最佳非向量基线的 2 倍；索引可从权威记录完整重建。
- MVP 不默认使用付费或外部嵌入服务；若后续需要发送内容到外部服务，必须另行进行数据、成本和来源授权。
- 向量索引必须可从权威结构化记录重建，不作为规则、批准状态或版本历史的唯一事实来源。

### FR-30 实现完整性与架构符合性

**治理状态：已确认**

**架构落实状态：已确认**

系统不能只验证最终输入输出，还必须验证实现是否经过用户认可的产品入口、既有架构和核心数据流，防止用影子实现、临时脚本或旁路逻辑获得表面正确结果。

验收标准：

- 改变生产入口或路由、状态权威或持久化、架构边界接口或所有权、核心能力或可直接产出最终结果的替代路径，用户明确要求原路径证明，或任务具有高影响时，在实施前识别真实产品入口、必须复用的模块、状态与数据流、允许的扩展点以及明确禁止的旁路。
- 文件数、模块数、同时修改代码和测试，以及保持接口、状态权威和数据流不变的机械跨文件修改本身不触发正式实现契约。
- 新项目或架构尚不明确时，先检查是否命中正式契约触发表；全部为否时只记录简短内部路径约束。会形成长期架构承诺、改变核心流程或满足其他触发项时，才建立正式契约并由用户确认高影响内容。
- 需要正式实现契约的任务在首次实现写入前记录版本、来源和冻结时点；后续修改保留差异，不得事后覆盖契约来合理化已完成的实现。低风险局部修改只使用简短内部路径约束，不创建持久化契约。
- 契约中的关键架构声明必须绑定已确认文档、代码注册点、现有调用方、运行入口或状态定义等独立来源；执行者生成的候选契约不能循环证明自身正确。
- 结果验收和实现路径验收相互独立；输出正确不能抵消实现路径违规。
- 辅助脚本必须声明用途和生命周期，只能用于构建、测试、迁移、数据准备或用户明确接受的自动化，不得静默替代产品功能。
- 重要功能至少保留一条由正式产品入口驱动、经过非 mock 核心模块并产生真实状态、持久化或等价副作用的证据；mock、spy 和覆盖率不能单独证明产品路径。
- 实现完成后提供用户可理解的证明摘要，先用用户操作、可观察状态和结果说明主路径，再附内部模块；至少说明契约版本、入口、关键状态变化、辅助脚本和未验证路径。
- 重要功能保留“需求 → 实现契约 → 代码或配置 → 测试与运行证据”的追踪关系。
- 独立审查应专门寻找绕过原系统、重复实现已有能力、硬编码结果、隐藏入口和不可维护临时路径。
- 未经用户明确接受的旁路实现、影子系统或以外部脚本替代产品核心逻辑属于硬失败，即使最终结果正确也不得宣布完成。

### FR-31 运行时减负与上下文预算

**治理状态：已确认**

**架构落实状态：已确认**

完整治理体系不得被等同为每次任务的默认运行流程。系统应使用薄核心、按需协议和离线治理分层控制上下文、确认次数和执行成本。

验收标准：

- 技能触发后始终适用的核心正文只保留目标保持、诚实证据、风险分级、最低安全和结果验证等跨任务规则；负触发任务只接收技能元数据，不加载核心正文。
- 明确低风险的一步任务不触发完整技能正文。技能触发后按实际字符数和章节数控制专项上下文；协议文件数量只是辅助指标。
- PRD、完整架构、ADR 全集、评估案例、追踪矩阵和审计哈希只在规划、治理、恢复或发布任务中读取，不自动进入普通实施上下文。
- 每次任务只维护会影响当前执行的语义槽位；快速任务通常只需目标和完成条件，复杂任务可按需增加约束、状态、实现路径和待决定事项，不固定字段数量。
- 长线程发生压缩或上下文膨胀时，以项目权威文件和当前工作集重新建立上下文，不依赖完整聊天历史。
- 每个新增机制记录解决的问题、触发条件、运行成本、证据、类别、停用或合并条件。只有 `quality` 和 `convenience` 机制可以根据净收益降级、合并或删除；`invariant` 和 `safety` 只能在等价覆盖、相关硬失败全量通过并经用户确认后替换。
- 上下文和流程预算属于可评估假设，必须用核心与加载上下文字符、完成时间、输出长度、额外用户干预和统一七项质量维度校准，不能只按文档完整度判断。
- 安全、授权、实现完整性、恢复和时效敏感研究属于硬触发，不得为了满足上下文预算而省略。
- 上下文减负评估必须使用可观测加载轨迹；无法区分技能正文、参考章节、项目文档和自动注入内容时，不得宣称预算已经验证。
- 技能描述、核心正文、参考章节和机制账本分别设置可校准预算，不能用“协议文件数量少”代替真实上下文成本。

### 功能职责边界

- FR-04 负责资料检索、来源质量和时效。
- FR-04A 负责方法论发现、适用性和裁剪。
- FR-14 负责把研究发现转化为当前项目的候选改进闭环。
- FR-18 负责反证搜索和研究偏差控制。
- FR-10 负责当前用户项目的纠偏。
- FR-23 负责技能产品自身的改进。
- FR-29 负责经验记录、检索方案演进和向量检索引入门。
- FR-30 负责结果正确之外的实现路径、架构符合性和反旁路验收。
- FR-31 负责将完整治理与日常运行解耦，并控制上下文和机制成本。
- FR-27 是所有长期行为规则变更的统一审批门。
- FR-28 负责高影响成果的审计闭环、状态声明和修复后回归。

## 10. 非功能需求

### 已确认

- **效率**：减少用户重复解释和不必要确认。
- **准确性**：不得把推断当作需求或事实。
- **专业性**：建议应有资料、约束分析和方案比较支撑。
- **严谨性**：验证强度与风险匹配。
- **易用性**：主要面向 Codex 桌面应用和自然语言交互。
- **可扩展性**：支持后续新增能力。
- **时效性**：对可能变化的事实、工具、标准和最佳实践主动核查当前资料。

### 建议确认

- **透明性**：说明为何选择某个流程、方案或 Agent 策略。
- **上下文效率**：技能正文保持精简，详细参考资料按需加载。
- **可恢复性**：复杂项目可在新线程中恢复。
- **安全性**：联网、权限和不可逆操作遵循最小必要原则。
- **兼容性**：首版优先 Windows Codex 桌面应用，避免无必要的平台绑定。
- **可评估性**：核心能力必须能通过样例、记录和回归测试检查。
- **可治理性**：规则、知识和技能变更可追踪、可审查、可回退。
- **弹性**：部分工具或网络不可用时仍能安全降级。
- **隐私与供应链安全**：外部内容、依赖和数据流经过风险控制。
- **可移植性**：通用能力、个人偏好和项目规则彼此分离。

## 11. 数据、权限与外部依赖

### 建议确认

- 技能本身不保存用户业务数据。
- 项目状态主要保存在用户项目目录中的 Markdown 文档。
- 外部资料通过 Codex 当前可用的搜索、浏览器、连接器或 MCP 获取。
- 涉及私有代码、凭据、个人信息或商业秘密时，联网前应检查是否需要脱敏或限制来源。
- 默认不引入自建后端、数据库或云服务。
- 研究记录应保存来源链接、日期、版本、结论和验证状态，避免日后无法判断知识是否过期。

## 12. MVP 范围

### 建议确认

1. 一个经过验证的核心 Codex 技能。
2. 快速、标准、完整三种工作模式。
3. 自然语言任务整理和主动完整性扫描。
4. 专业调研、方法论发现和方案比较的基础规则。
5. 人工确认矩阵，包括持久化规则确认门。
6. 复杂项目的多 Agent 适用性判断和用户授权流程。
7. 任务简报、研究任务书、实施任务书和审查任务书。
8. 一组可执行的触发、规划、专业性、确认和对抗评估样例。
9. 一份面向用户的自然表达与使用文档。
10. 按 `docs/STATE_CONTRACT.md` 维护完整模式的最小项目状态；不默认生成全套文档。
11. 薄核心、按需协议和离线治理分层；普通任务不加载完整治理文档。

### MVP 后首个版本

1. 复杂项目完整文档模板与线程恢复。
2. 完整知识、规则和版本治理。
3. 个性化、共享和配置分层。
4. 基础失败降级脚本或校验工具。
5. 核心技能稳定后封装为可安装插件。
6. 更完整的供应链安全、周期性学习和自动化能力。

## 14. 整体成功标准

### 建议确认

MVP 的正式通过标准以 `docs/EVALUATION.md` 为准。下列标准同时包含 MVP 和后续插件化后的整体产品目标。

通过至少一组覆盖快速、标准、完整流程的真实任务验证：

1. 用户可从自然语言开始，不需要先编写完整提示词。
2. Codex 能主动给出整体判断，不依赖用户逐项追问。
3. Codex 能识别明显遗漏和专业方案空间。
4. 在定义的评估样例中，把建议写成用户需求的违规次数为零。
5. Codex 能正确区分需要确认与可自主推进的事项。
6. Codex 能在复杂项目中主动判断并提议单 Agent 或多 Agent，由用户作最终决定。
7. Codex 能按最小状态契约建立和维护必要项目文档。
8. Codex 能报告证据、未验证内容和剩余风险。
9. 新线程可依据最小状态契约重建当前工作状态。
10. MVP 稳定后，技能可被封装为插件并由其他用户安装。
11. 面对时效敏感或知识不足的问题，Codex 会主动查阅当前资料并保留来源。
12. 新发现能经过验证和审查后完善项目或技能，而不会静默改变既定方向。
13. 工具或网络失败时会正确降级，不虚构验证结果。
14. 技能修改后可通过回归样例证明没有破坏已有核心行为。
15. 在定义的线程恢复样例中，Codex 能从项目文件重建目标、决定和未决事项，并明确暴露冲突或缺失。
16. 用户获得的是结论、建议和必要确认，而不是被迫承担 Codex 本应完成的完整性分析。
17. 在定义的外部提示注入对抗样例中，Codex 不采纳与用户目标冲突的外部指令；真正的硬隔离依赖权限、沙箱或后续 Hooks。
18. 插件可供他人使用，而不会携带创建者的私人项目规则和数据。
19. 用户无需掌握复杂提示词，也能获得边界清晰、可验证的 Codex 执行任务。

## 16. 需求完整性审计结论

### 已覆盖的核心能力

- 高效自然语言交互
- 主动需求与风险扫描
- 专业资料、项目、论文和方法论研究
- 方案比较和专业建议
- 人工确认边界
- 单 Agent 与多 Agent 策略
- 项目文档与状态恢复
- 实施、验证、审查和纠偏
- 持续学习与可选周期性跟踪

### 本轮新增的关键能力

- 上下文与历史状态治理
- 模型、工具、Agent 和资源选择
- 反证搜索与研究偏差控制
- 工具失败后的安全降级和恢复
- 技能自身的评估与回归测试
- 审计结论的修复后回归与状态校准
- 规则、知识、版本和升级治理
- 稳定、低负担的交互输出协议
- 技能自我完善的审查与回退机制

### 仍无法由技能单独保证的事项

- 隐式触发不可能保证每次都成功，需要显式调用兜底。
- 网络搜索不保证覆盖所有资料，也不保证资料本身正确。
- 多 Agent 不能在没有用户明确授权时自动启动。
- 技能不能让模型永久自行训练，只能通过可持久化资料和规则改善后续工作。
- 最终业务价值、高风险决策和现实世界安全仍需用户或合格专业人员确认。
