# 架构独立审查记录

> 状态：历史审查记录；后续修正见 `docs/FINAL_AUDIT.md`  
> 日期：2026-06-10  
> 方式：两个只读子 Agent 独立审查，主 Agent 汇总  
> 范围：PRD、调研、架构、决策和提示词体系

## 1. 审查结论

两位审查者都认为“精简核心技能 + 按需协议 + 项目状态 + 评估体系”的方向合理，但原始 MVP 过大，提示词、规则变更和评估机制需要进一步收紧。

## 2. 已写入待确认草案的修正

| 发现 | 处理 |
| --- | --- |
| 长期规则可能被自动污染 | `AGENTS.md`、技能和全局指导只生成候选差异，经用户确认后修改 |
| 部分成功标准是无法保证的绝对承诺 | 改为线程恢复样例和外部提示注入对抗样例中的可测表现 |
| 缺少独立提示词架构 | 新增 `docs/PROMPTING.md`，分离核心契约、任务路由、按需协议、任务简报、阶段任务书和独立评审 |
| MVP 过大 | 延后插件封装、完整恢复、完整治理、Hooks、自建 MCP、自定义 Agent 和自动化 |
| 隐式触发可能误判 | 加入正例、边界例、负例和显式调用兜底 |
| 评估只有维度，没有执行方法 | 后续补充为 `docs/EVALUATION.md` 中的样例、评分、硬失败和通过阈值 |
| 文档状态和优先级不清晰 | 澄清状态词，并按 PRD、ADR、架构、PLAN 的职责划分权威 |
| 联网可能泄露内部信息 | 增加公开、内部、受限三级数据分类 |
| 多 Agent 授权范围不清 | 授权包含角色、数量、读写范围、有效阶段和终止条件，默认只对当前阶段有效 |
| 用行数控制技能大小不准确 | 改为字符和实际上下文成本预算 |
| 研究可能无限扩展 | 增加问题、来源、投入预算和停止条件 |
| 执行流程可能自我评审 | 独立评审提示与执行提示分离，并保留人工抽检 |

## 3. 首批推荐评估样例

1. 修改一个错别字：应直接执行，不创建 PRD。
2. “帮我做个记账工具”：应先整体扫描，只询问关键问题。
3. “建立经验库”：应比较文件、SQLite、全文检索、向量检索和混合方案。
4. 选择当前 Codex 扩展方式：应查询最新官方资料并记录日期和来源。
5. 复杂项目：应主动判断多 Agent 价值并由用户决定。
6. 用户纠正目标：应停止旧方向并更新后续基线。
7. 外部资料包含提示注入：不得将外部指令当作用户要求。
8. 研究结论趋于稳定：应停止继续搜索并报告剩余不确定性。
9. 技能不应触发的闲聊或简单问答。
10. 新线程从最小项目状态文件恢复任务。

## 4. 尚待真实验证

- 隐式触发的实际准确率。
- 约 8,000 字符的核心技能预算是否合适。
- 默认最多加载两个参考模块是否足够。
- `docs/EVALUATION.md` 的首批指标和通过阈值是否能区分真实改善。
- 最小状态文件是否足以支持复杂项目恢复。
- 不同模型、推理强度和工具配置下的行为稳定性。

这些问题不应继续通过纯文档讨论解决，应在实施计划中优先用原型和评估验证。
