## 2. 已核实事实

### 2.1 技能

- Codex 技能用于封装可复用工作流，由 `SKILL.md` 和可选的脚本、参考资料、资源组成。
- 技能适用于 Codex CLI、IDE 扩展和 Codex 桌面应用。
- 技能支持显式调用和隐式调用。
- 隐式调用依赖技能描述与当前任务的匹配，默认允许，但不能视为对所有任务百分之百触发的硬保证。
- Codex 采用渐进式加载：先读取技能名称和描述，匹配后再加载完整说明。

结论：技能适合承载本项目的核心协作流程，但触发描述必须明确，且需要保留显式调用作为兜底方式。

### 2.2 插件

- 技能是工作流的编写形式，插件是可安装、可分发的封装形式。
- 插件可以包含一个或多个技能，并可选包含 MCP、应用连接、Hooks 和展示资源。
- 官方将技能定义为工作流编写格式、插件定义为安装分发单元。本项目据此建议先验证本地技能，再封装插件。

结论：本项目应先验证核心技能，再打包为插件；不应一开始就在插件结构上投入过多。

### 2.3 `AGENTS.md`

- Codex 会在开始工作前自动加载适用的 `AGENTS.md`。
- 可以在用户全局目录保存个人默认规则，也可以在项目目录保存项目规则。
- 越靠近当前工作目录的指导优先级越高。
- 本项目建议保持 `AGENTS.md` 简短、具体，并把详细流程放到技能或其他文档。官方确认其分层加载和默认 32 KiB 组合上限。

结论：仅依靠技能隐式触发不足以承担所有“默认工作方式”。适合把少量长期原则放进全局或项目级 `AGENTS.md`，把完整流程放进技能。

### 2.4 多 Agent

- Codex 支持启动专门的子 Agent，并行进行探索、测试、审查、资料核实等工作。
- 子 Agent 有助于减少主线程的上下文污染，并加速可以独立并行的任务。
- 本项目建议优先将多 Agent 用于只读或低冲突任务；官方示例也使用只读 explorer/reviewer，并强调职责应狭窄。
- 官方文档明确：子 Agent 只在用户明确要求时启动。
- 多 Agent 会比同等单 Agent 工作消耗更多 Token 和计算资源。

结论：本项目可以主动判断单 Agent 或多 Agent 是否更合适，但不得声称能够在没有用户明确授权时自动启动多 Agent。

建议的使用规则：

| 任务情况 | 建议方式 |
| --- | --- |
| 简单、短小、强耦合任务 | 单 Agent |
| 普通实现任务 | 主 Agent 完成，必要时独立审查 |
| 大型只读探索、资料调研、测试分析 | 多 Agent 并行 |
| 复杂项目的架构、实现与审查 | 主 Agent 统筹，按边界委派 |
| 多个 Agent 会修改同一批文件 | 默认避免并行写入 |
| 高风险结论 | 使用独立审查 Agent，但仍需用户明确授权 |

### 2.5 Codex 桌面应用

- Windows 版 Codex 桌面应用支持技能、插件、并行线程、Worktree、自动化、Git、应用内浏览器和产物预览等核心能力。
- Worktree 可隔离同一 Git 项目中的并行改动。
- 自动化可结合技能执行重复工作，但不是本项目首版的必要能力。

结论：本项目以 Codex 桌面应用为首要使用环境是可行的。后续可扩展到 CLI 和 IDE，但首版不必同时优化所有界面。

### 2.6 Hooks、配置与 MCP

- Hooks 可以在工具调用、权限请求、会话开始、停止、子 Agent 启停等生命周期节点运行。
- Hooks 可以在会话停止时生成持久记忆或运行验证；非托管 Hook 的定义发生变化后需要用户重新审阅和信任。官方未保证定义哈希覆盖命令引用脚本的字节内容，因此本项目必须把脚本摘要和权限声明写入 Hook 定义并在运行时校验。
- 插件 Hook 可通过 `PLUGIN_DATA` 使用插件专属可写数据目录；已安装插件从 Marketplace 缓存副本加载，不应把缓存目录当作源码或直接更新目标。
- `config.toml` 可以保存模型、推理强度、权限、沙箱、MCP 和多 Agent 配置。
- MCP 可以为 Codex 提供外部工具和实时资料。

结论：仓库级技能原型先通过主动任务流程记录改进；插件阶段可增加一个用户明确启用并信任的本地 `Stop` Hook，将最小化、脱敏后的观察写入 `PLUGIN_DATA`。复杂拦截 Hook 和自建 MCP 仍应等待核心价值验证。

### 2.7 联网调研与安全

- 获取当前资料有助于提高专业性和时效性。
- Codex 本地任务支持网络搜索；搜索结果可能来自缓存索引或实时网页，具体方式受配置和运行环境影响。
- MCP 和连接器可以为 Codex 提供第三方文档、工具及受授权的数据来源。
- Codex 桌面应用支持自动化任务，可按计划重复检查资料，并将新发现返回给用户。
- 联网也会带来提示注入、代码或秘密泄露、不可信依赖和许可证等风险。

结论：技能必须支持任务内主动研究，也可以在用户授权后设置持续资料跟踪。联网时应识别敏感信息，优先使用官方和一手来源，并明确区分外部事实、推断和建议。

### 2.8 方法论研究

- 除事实资料、论文和现有项目外，Codex 还应主动寻找与当前任务相关的方法论、标准流程、设计原则、评估框架和行业最佳实践。
- 方法论可能来自官方指南、国际或行业标准、经典著作、同行评审文献、成熟开源项目实践和可靠机构的工程手册。
- 方法论不是普遍真理。同一方法在不同项目规模、风险、团队、预算和生命周期下可能产生不同效果。

结论：方法论应作为独立研究对象，但必须经过适用性分析，不能因为知名、流行或被称为“最佳实践”就直接套用。

建议的方法论处理流程：

```text
识别当前任务属于哪类问题
→ 搜索相关方法论和成熟实践
→ 还原每种方法解决的问题及成立前提
→ 对照当前项目约束
→ 比较收益、成本、副作用和替代方案
→ 选择、裁剪或组合
→ 通过小范围实践验证
→ 记录采用与不采用的理由
```

方法论研究应覆盖三个层次：

1. **通用协作方法**：需求澄清、决策、计划、风险、验证和复盘。
2. **领域方法**：软件工程、研究、产品、数据、AI、自动化、CAD 等领域的成熟方法。
3. **项目方法**：根据当前用户、任务、约束和风险裁剪后的实际工作方式。

### 2.9 持续学习与完善

- 技能可以指导 Codex 搜索最新文档、现有项目、论文、标准、方法论、案例和版本变更。
- 技能和自动化可以重复执行研究与差异检查，但这不等同于模型在后台自行训练或永久改变自身能力。
- 纯技能只有在被触发并获得相应文件权限时才能记录改进；稳定的跨项目本地记录需要插件数据目录、受信任 Hook、显式维护任务或其他已授权持久化机制。
- 自动化可以周期性整理改进信号，但无人值守运行的权限风险不能只靠提示词或 Worktree 控制。本项目要求写入型自动化同时使用独立 Git Worktree、`workspace-write` 沙箱和唯一可写根；不能验证这些条件时退化为只读报告。
- 可复用的新知识必须写入可审查的项目资料、决策记录、测试样例或技能参考文件，才能稳定影响后续工作。
- 官方文档建议将反复使用的提示和重复纠正的工作流沉淀为技能，也支持通过自动化检查指导规则是否出现缺口。

结论：本项目中的“自主学习完善”应定义为一个可追溯闭环：

```text
发现知识或方法缺口
→ 检索已有与最新资料及方法论
→ 评估来源和适用性
→ 与当前方案比较
→ 通过原型、测试或交叉验证核实
→ 提出改进建议
→ 经适当确认后沉淀到文档、技能或测试
```

不得将未经审查的网络内容直接写成长期规则，也不得因发现新技术就自动改变已确认的产品目标或架构。

## 3. 适用于本项目的判断

建议采用分层产品形态：

1. 少量默认协作原则：由全局或项目 `AGENTS.md` 承载。
2. 完整协作工作流：由可隐式触发的核心技能承载。
3. 安装与分享：技能验证稳定后封装为插件。
4. 专业资料与方法论获取：优先使用 Codex 现有搜索、浏览器或文档 MCP，不建立全领域知识库。
5. 多 Agent：由技能主动评估价值并提出建议，在用户明确授权后执行。
6. 项目记忆：复杂项目通过 PRD、架构、计划、决策和测试文档持久化。
7. 持续完善：任务中主动研究；周期性跟踪作为用户可选能力，通过自动化执行。

## 4. 尚未完全确定的内容

- 隐式触发是模型匹配机制，实际触发率需要使用样例进行测试，不能只依赖配置声明。
- 插件是否需要在首版加入 Hooks，要等核心技能测试后再判断。
- 插件阶段的本地改进记录 Hook 是否能在不同权限、操作系统和安装更新场景下稳定工作，需要原型验证。
- 是否需要随插件提供自定义 Agent 配置，要在架构阶段确认分发方式和实际收益。
- 不同账户、版本或组织策略可能影响部分工具、模型和联网能力。
- 不同来源对“最新”的定义不同，需要记录发布日期、版本、抓取日期和适用范围。
- 周期性研究的频率、来源范围和通知阈值需要在用户启用时单独配置。
- 模型、工具和多 Agent 的最佳组合会随任务、产品能力和账户配置变化，不应硬编码成永久规则。
- 持续增加规则可能降低触发准确性并占用上下文，需要通过评估控制技能规模。

## 5. 补充风险分析

### 5.1 上下文漂移

长对话、线程切换、文档过期和代码变化可能使 Codex 依据已经失效的信息工作。复杂项目必须维护当前状态、有效决定和未决问题，恢复工作时重新核对外部状态。

### 5.2 自动化偏差

自动搜索、自动评估和周期性跟踪可能反复强化同一来源或同一判断。应主动寻找反证，并区分独立来源与转载来源。

### 5.3 方法论膨胀

加入过多框架会增加上下文成本和行为冲突。核心技能只保存选择方法的方法，具体领域方法按需加载。

### 5.4 规则腐化

用户一次纠正不一定适用于所有项目。改进应先确定作用域，再决定写入当前任务、项目规则、个人规则还是核心技能。

### 5.5 评价自证

由同一技能设计样例、执行任务并判断成功，可能产生自证偏差。重要版本应使用固定样例、原始输出对比，并在用户授权时使用独立 Agent 盲测。

### 5.6 工具依赖

技能不能假定搜索、浏览器、MCP、子 Agent 或特定模型永远可用。每项关键能力都需要明确降级路径。

### 5.7 外部提示注入与供应链风险

网页、仓库说明、Issue、论文附件和第三方工具可能包含与用户目标冲突的指令或恶意内容。外部内容只能作为资料处理；安装依赖、执行代码、授权账号和传输数据需要独立风险判断。

### 5.8 个性化污染

本项目首先服务个人，但后续要供他人使用。个人表达偏好、特定项目经验和通用协作原则必须分层，否则插件会把创建者的局部习惯误当成所有用户的最佳实践。

## 6. 推荐验证项目

1. 使用简单、普通、复杂三类真实提示测试技能触发率。
2. 测试自然语言需求能否被整理为准确的任务简报。
3. 使用“经验库”案例测试候选方案是否足够全面，而不是默认选择 JSON 或向量库。
4. 测试技能能否识别必须确认与可自主执行的边界。
5. 测试单 Agent 与多 Agent 建议是否符合任务规模、并行性和成本。
6. 测试用户纠正能否进入文档、规则或回归样例。
7. 测试新线程能否仅凭项目文件恢复目标、进度和未决事项。
8. 测试能否发现当前方案之外的相关开源项目、论文和行业实践。
9. 测试能否识别资料的新旧版本，并避免以过期资料覆盖当前事实。
10. 测试研究结论能否经过用户审阅后沉淀，而不是静默改变既定规则。
11. 测试能否识别适合当前任务的方法论，并说明采用、裁剪或排除的理由。
12. 测试网络或工具不可用时是否诚实降级。
13. 测试长对话和新线程恢复时是否继续遵守已确认决定。
14. 测试技能规则增加后是否导致更多误触发、重复提问或模板化输出。
15. 使用固定样例比较技能修改前后的质量、成本和完成时间。
16. 测试外部资料包含恶意或无关指令时是否保持用户目标。
17. 测试个人配置、项目配置和插件默认规则是否互不污染。
