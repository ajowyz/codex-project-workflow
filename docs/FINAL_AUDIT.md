# 架构门最终审计

> 状态：历史时点审计记录；原“具备集中确认条件”结论已于后续流程复盘中撤回  
> 日期：2026-06-10  
> 方式：主 Agent、本地一致性扫描、两位只读子 Agent、OpenAI 官方文档复核

## 1. 结论

未发现需要推翻推荐方案的 P0 问题。审计发现的 P1 矛盾已经写入文档修正。此处曾据此判断架构具备集中确认条件；后续用户要求再次检查时又发现新的问题，说明该判断缺少“修复后回归、反方挑战和状态声明证据”约束，因此原结论撤回。

## 2. 已关闭问题

| 问题 | 修正 |
| --- | --- |
| MVP 与插件交付冲突 | MVP 只验证仓库级技能和文档，插件封装移至下一版本 |
| 旧总结与当前规则冲突 | 根文档标记为历史背景，不作为执行基线 |
| 长期规则可能自动污染 | 普通文档可维护；`AGENTS.md`、技能和全局规则只能生成候选补丁，评估并经用户批准后生效 |
| 最小状态与完整恢复混淆 | 新增 `docs/STATE_CONTRACT.md`，MVP 仅承诺最小状态重建 |
| 评估无法判定通过 | 新增 `docs/EVALUATION.md`，定义评分、硬失败、阈值、次数和首批样例 |
| 快速任务仍承担完整流程 | 改为先做不可见轻量分流，快速模式直接执行，不显示模板、不加载参考模块 |
| 隐式触发不可测试 | 增加候选 `description` 和 10/10/10 正负边界触发测试 |
| 联网分级缺少操作协议 | 增加联网预检、脱敏查询和首次联网确认 |
| 多 Agent 边界不稳定 | 增加复杂性判定条件和授权状态机，授权默认只对当前阶段有效 |
| 技术决定可能覆盖产品需求 | 改为按职责划分文档权威，ADR 影响 PRD 时必须同步更新和重新确认 |
| 研究可能无限扩展 | 为快速、标准、完整模式增加软预算和停止条件 |
| 独立审查与用户拒绝多 Agent冲突 | 定义单 Agent 反方审查、新上下文复核、子 Agent 复核三级方式 |
| 核心技能 8,000 字符被误作官方限制 | 明确这是内部实验预算；官方 8,000 字符只针对初始技能列表名称和描述 |
| 回滚依赖不存在的 Git | 增加实施前初始化 Git 或建立快照的前置决定 |
| 状态与阶段说明过期 | 统一状态词并更新 PRD 当前阶段 |

## 3. 仍需通过原型验证

1. 候选技能描述能否达到正触发 9/10、负触发 9/10、边界 8/10。
2. 快速模式的耗时和输出长度能否控制在无技能基线增加 20% 以内。
3. 三个 MVP 参考模块是否足够，是否仍存在提示词过载。
4. 评估评分能否稳定区分改善和随机波动。
5. 最小状态契约能否在新线程中达到关键字段 100% 正确。
6. 联网预检能否阻止受限数据进入查询。
7. 多 Agent 提议是否既能主动出现，又不会过度打断。
8. 仓库级技能能否在 Codex 桌面应用中可靠加载和更新。

这些是实施计划的最高风险验证项，不应继续靠文档推测。

## 4. 后续流程复盘

本次审计本身暴露了一个新的 P1 流程缺陷：

- 初次审查和问题修复完成后，直接把“当前检查未发现更多阻塞”升级成“架构已经达到可以确认的状态”。
- 修复前的审计结论没有被自动视为失效。
- 第二次检查仍能发现新问题，说明第一次采用的是补缺式检查，不是以推翻结论为目标的反方检查。

已写入候选架构的修正：

- `docs/ARCHITECTURE.md` 最初增加审计闭环与五级状态声明；后续反方复核又将其修正为治理状态和证据状态两条独立轴。
- `docs/EVALUATION.md` 增加硬失败条件和 E13 回归样例。
- `docs/PROMPTING.md` 增加状态词校准与反证检查。
- `docs/DECISIONS.md` 增加 ADR-022。

本文件只保留该历史时点。当前架构状态、后续修正和是否达到待确认候选，以 `docs/PLAN.md` 和最新的版本绑定审计记录为准。

## 5. 官方复核

截至 2026-06-10：

- 技能可显式或按 `description` 隐式调用，完整 `SKILL.md` 在选择后加载。
- 初始技能列表的约 8,000 字符预算不是完整 `SKILL.md` 的限制。
- 插件是安装分发单元，可包含技能、Hooks、MCP、应用和资产。
- `AGENTS.md` 分层加载，默认组合上限为 32 KiB。
- 子 Agent 只在用户明确要求时启动，并增加 Token 成本。
