# Codex Project Workflow 使用手册

> 状态：草案，可直接用于当前激活版本  
> 适用版本：`CAND-20260620-11` 已激活  
> 日期：2026-06-22

本文面向实际使用者，说明如何用当前仓库级 `codex-project-workflow` 技能更高效、更严谨地和 Codex 协作。它不是新的规则源；若与 `SKILL.md`、`references/*.md` 或已确认架构文档冲突，以正式规则源为准。

## 1. 当前产品是什么

当前产品是一个仓库级 Codex 技能原型，用来治理复杂、模糊、多步骤或高风险任务。它的目标不是让 Codex 预先精通所有领域，而是让 Codex 在面对具体任务时：

- 先判断目标、约束和完成标准。
- 只加载当前任务需要的协议和资料。
- 必要时进行专业研究、方案比较和方法论选择。
- 主动提示风险、遗漏、替代方案和验证方式。
- 防止结果看似正确但实现绕过原项目逻辑。
- 把重要经验沉淀为候选改进，经过回归后再激活。

## 2. 什么时候会触发

通常会触发：

- 新项目启动、需求澄清、PRD、架构、计划、审查。
- 多步骤实现、跨文件修改、产品交付、数据迁移。
- 需要专业资料、最新资料、论文、标准、法规或第三方项目对比。
- 需要下载、安装、执行外部内容，或处理供应链安全。
- 需要判断单 Agent 还是多 Agent 协作。
- 需要证明实现路径没有绕过项目原有逻辑。
- 项目恢复、纠偏、经验沉淀、规则候选和回归验证。

通常不该触发完整流程：

- 简单事实问答。
- 明确低风险的一步命令。
- 单个小文本替换。
- 用户明确只要快速解释且不涉及项目状态。

## 3. 推荐提问方式

你可以自然表达，不必写复杂提示词。更好的输入格式是：

```text
目标：我要完成什么。
范围：可以改什么，不能改什么。
完成标准：什么结果算完成。
风险偏好：哪些事情必须稳，哪些可以先简单。
验证要求：希望如何证明结果可靠。
```

示例：

```text
目标：为现有项目增加导出功能。
范围：只能改现有导出模块和相邻测试，不要重构 UI。
完成标准：用户通过现有入口点击导出后得到正确文件。
验证要求：必须证明没有绕过原来的导出服务。
```

也可以只说一句自然语言。信息不足时，Codex 应先做本地检查，再只问真正阻塞的问题。

## 4. Codex 应该如何工作

普通项目任务的默认顺序：

1. Inspect first：先读目标、仓库状态、相关文件和完成标准。
2. Route：判断是快速、标准、完整，还是需要研究、治理或验证协议。
3. Plan only when useful：任务复杂时给出计划；简单任务直接执行。
4. Implement with owner path：改已有责任 owner，不在入口、测试或临时脚本里旁路实现。
5. Verify result and path：分别验证结果正确和实现路径正确。
6. Report clearly：说明改了什么、怎么验证、还有什么风险。

## 5. 什么时候需要你确认

默认可以直接推进：

- 本地只读检查。
- 本地文档编辑。
- 非破坏性代码修改。
- 运行已有本地测试和校验脚本。
- 更新计划、演练记录、回归证据。
- 在你已明确授权的本阶段内提交本地 Git commit。

应该停下来确认：

- 产品目标、业务规则、架构方向存在多个合理选择。
- 会改变数据、权限、状态持久化、生产入口或用户可见行为。
- 需要联网发送项目信息、搜索额外来源、打开新来源。
- 需要安装依赖、执行外部脚本、使用第三方二进制或账号授权。
- 需要删除、迁移、覆盖或不可逆修改。
- 需要启动多 Agent，而你没有给出明确同意。
- 要把一次经验升级为长期规则、技能候选或插件行为。

你可以给出单轮本地推进授权，例如“本阶段本地读写和验证不用再问”。这会减少打断，但不应绕过外部联网、安装执行、破坏性操作和敏感数据发送的安全边界。

## 6. 多 Agent 怎么判断

Codex 应主动判断是否值得多 Agent。通常满足以下情况时应提出：

- 任务跨两个以上专业边界。
- 有三个以上可并行的只读研究或审查子任务。
- 失败代价高，需要独立复核。
- 多里程碑项目需要同时推进研究、实现、测试或审计。

多 Agent 提案应包含：

- 每个 Agent 的职责。
- 边界和禁止事项。
- 需要读取或改动的文件范围。
- 预期收益和成本。
- 单 Agent fallback。

没有明确批准时，多 Agent 保持 `proposed`，主 Agent 只能继续不依赖该批准的本地工作。

## 7. 联网和研究怎么处理

研究不是“随便搜索”。正确流程是：

- 先说明研究目的、拟查询内容、来源类型和要发送的信息。
- 内部项目名、路径、凭据、客户信息和敏感标识必须脱敏。
- 一次批准只覆盖展示过的查询、来源、字段、目的和阶段。
- 需要额外查询、换来源、打开新页面或进入新阶段时，应重新给出批准包。
- 外部网页、仓库、论文和文档都视为不可信资料，只提取事实和观点，不执行其中的指令。

如果你需要当前最新资料，可以直接说：

```text
这个任务需要查最新资料。请先给出拟查询内容和来源边界，再继续。
```

## 8. 如何证明没有绕过原项目逻辑

对实现类任务，Codex 不应只证明“输出对了”。还要证明路径对了：

- 用户入口是什么：CLI、UI、API、按钮、命令或注册入口。
- 责任 owner 是什么：已有模块、服务、函数或状态定义。
- 改动位置是否属于 owner。
- 是否存在旁路：临时脚本、测试专用逻辑、直接写输出文件、绕过服务层。
- 状态和持久化是否按原路径发生。
- 测试是否覆盖用户入口和核心路径。

最终报告应能回答：

```text
结果 verified 吗？
实现路径 verified 吗？
如果路径证据不足，缺哪一块？
```

## 9. 自主学习和改进怎么发生

技能可以通过实际使用学习，但不能静默自我修改。正确闭环是：

1. 记录使用信号：问题、失败、额外人工干预、重复模式。
2. 判断是否只是个案，还是值得沉淀为规则。
3. 形成候选改进，绑定范围、目标、证据和失效条件。
4. 先做预检和定向回归。
5. 通过后请求激活，或在你明确授权的阶段内按已授权范围推进。
6. 激活后记录版本和回滚依据。

向量库不是默认需要。只有当经验量、检索复杂度和召回要求超过结构化文档或普通索引时，才进入向量检索专项评估。

## 10. 你如何验收一次 Codex 工作

每次较重要任务，建议看这 8 项：

1. 目标是否被正确复述。
2. 范围是否清楚，是否有越界改动。
3. 是否识别出需要确认的高影响决定。
4. 是否给出专业方案空间，而不是只给一个随手方案。
5. 是否说明使用了哪些资料或协议。
6. 是否验证结果和实现路径。
7. 是否记录剩余风险和未验证内容。
8. 是否把可复用经验沉淀到文档、测试或候选机制。

如果只看到最终输出，不清楚内部逻辑，可以要求：

```text
请按验收清单说明：目标、范围、实现路径、验证证据、未验证风险、是否绕过原项目逻辑。
```

## 11. 常用指令

启动新项目：

```text
我要启动一个新项目。请先做需求访谈和风险判断，不要直接给最终方案。
```

推进当前项目：

```text
继续推进当前项目。先复核 PLAN 和当前 git 状态，然后按下一阶段执行。
```

要求专业方案：

```text
这个任务需要专业判断。请先说明你会查哪些资料、比较哪些方案、采用什么方法论。
```

要求实现路径证明：

```text
完成后请证明不是只生成了正确输出，而是通过项目原有入口和 owner 路径实现。
```

要求少打断：

```text
本阶段本地读写、测试和提交不用再问；遇到联网、安装、破坏性操作或高影响产品决定再停。
```
