# 如何高效、严谨地使用 Codex 完成新项目

> 文档状态：历史讨论总结，仅作为背景参考，不是当前项目的需求或执行基线。  
> 当前权威文档：`docs/PRD.md`、`docs/ARCHITECTURE.md`、`docs/DECISIONS.md` 和后续 `docs/PLAN.md`。  
> 若本文与当前权威文档冲突，以当前权威文档为准。本文描述的完整流程主要适用于复杂或高风险项目；快速和标准任务应采用风险匹配的轻量流程。

> 适用范围：软件、AI、自动化、CAD、工程工具及其他需要持续开发的项目。  
> 核心原则：不要只让 Codex “写代码”，而要建立需求、研究、决策、实施、验证、审查和验收的完整闭环。

---

## 一、先建立正确预期

Codex 可以承担产品分析、资料研究、架构设计、编码、测试、审查和文档整理等工作，但它不是天然正确、全知且不会犯错的专家。

需要明确：

- Codex 不能自动掌握网上所有资料；重要问题必须有针对性地检索和核实。
- 检索到资料不等于理解了项目的真实目标。
- 模型可能把模糊需求自行补全，并把推断误当成事实。
- 代码可以运行，不等于产品方向正确。
- 测试通过，不等于业务、工程或安全要求正确。
- 如果代码、测试和验收标准都由同一个模型制定，可能出现“自己证明自己正确”。
- 对话记录不是可靠的项目管理系统，重要结论必须写入项目文档和测试。

因此，正确的协作关系是：

> 用户掌握最终目标和关键业务判断；Codex 主动研究、设计、实施、验证并反驳自己的方案。所有重要结论都要沉淀为需求、决策、代码、测试或验收记录。

---

## 二、新项目完整流程

新项目建议依次经过以下阶段：

1. 建立协作规则
2. 需求访谈和目标澄清
3. 专业资料调研
4. PRD 和验收标准确认
5. 架构设计和风险评审
6. 分阶段实施计划
7. 建立项目持久化文档
8. 按里程碑实施
9. 自动化验证和真实运行
10. 反方审查与独立复核
11. 用户业务验收
12. 变更控制、交付和维护

不要跳过前六个阶段直接要求 Codex 完成整个项目。

---

## 三、第一阶段：启动项目并建立协作规则

新项目第一次对话，可以直接发送：

```text
我要启动一个新项目。

你需要承担产品顾问、领域研究员、架构师、开发工程师、测试负责人和风险审查者的职责，但不能把自己的推断当成我的需求。

整个项目必须遵守：

1. 首要目标不是尽快增加代码，而是保证方向正确、事实可信、实现可验证、改动可回退。
2. 重要专业问题优先查阅官方文档、标准、论文或可靠的一手资料，不仅依赖模型记忆。
3. 严格区分：
   - 已核实事实
   - 专业推断
   - Codex 建议
   - 待确认假设
4. 主动寻找需求遗漏、边界条件、反例、失败方式、安全风险和长期维护成本。
5. 不得自行决定会影响产品目标、业务行为、数据、权限、架构或工程安全的模糊需求。
6. 不得因为代码存在、示例通过或文档声明，就认定能力已经可靠完成。
7. 用户的每次纠正必须进入需求文档、决策记录或回归测试，不能只留在聊天里。
8. 发现方向偏差时立即停止扩展并报告，不得用更多代码掩盖问题。
9. 没有充分证据时必须明确说“不确定”或“尚未验证”。
10. 未经我确认需求和架构，不开始正式编码。

现在先进入需求访谈，不要直接给出最终方案。
```

这段话的作用是明确：Codex 不是只负责执行，而是必须主动发现问题，同时不能擅自替用户决定关键需求。

---

## 四、第二阶段：需求访谈

让 Codex 主动向你提问，而不是让你独自写一份可能不完整的需求。

```text
请对我进行结构化需求访谈。

需要确认：

1. 项目真正要解决的问题。
2. 目标用户和实际使用场景。
3. 用户目前如何处理这个问题。
4. 核心流程和最高频任务。
5. 项目成功的可衡量标准。
6. 必须具备的功能。
7. 明确不做的内容。
8. 性能、稳定性、安全、权限、隐私和兼容性要求。
9. 输入、输出、数据来源和外部系统。
10. 异常、失败、恢复和边界场景。
11. 时间、预算、设备、技术和人员限制。
12. 哪些决定必须由我确认。

不要一次提出大量零散问题。请按优先级分轮访谈，每轮先问最影响项目方向的内容。
当我的想法存在错误、风险或不必要复杂度时，请直接指出并解释原因。
```

访谈期间，你需要重点确认：

- 谁会使用项目？
- 他们最重要的三个任务是什么？
- 什么结果才算成功？
- 什么错误绝对不能接受？
- 第一版最小范围是什么？
- 哪些内容明确推迟？

---

## 五、第三阶段：专业调研

涉及技术选型、外部 API、标准、法律、安全、机械设计、CAD、医学、金融或较大投入时，要求 Codex 先调研。

```text
在提出方案前，请先对关键专业问题进行调研。

要求：

1. 优先使用官方文档、标准、厂商资料、论文和可靠的一手资料。
2. 核对资料的版本、发布日期和适用范围。
3. 重要结论尽量交叉验证。
4. 说明资料中的通用方案是否适合本项目约束。
5. 明确区分资料事实、你的推断和你的建议。
6. 提供来源链接或准确出处。
7. 不要只罗列搜索结果，必须形成自己的判断。
8. 对无法核实的内容明确标记不确定性。

最终给出：
- 已核实事实
- 适用于本项目的结论
- 存在争议或不确定的内容
- 推荐方案及原因
- 需要通过原型或测试验证的假设
```

不要用“你是专家，直接给最好的方案”代替调研要求。专业性来自证据、推理和验证，而不是语气上的自信。

---

## 六、第四阶段：形成 PRD 和验收标准

需求访谈和调研完成后，要求 Codex形成正式需求基线：

```text
根据已经确认的访谈和调研结果，整理项目 PRD。本阶段仍不编码。

PRD 必须包括：

1. 背景和问题定义
2. 项目目标
3. 非目标
4. 目标用户
5. 核心使用场景
6. 用户流程
7. 功能需求
8. 非功能需求
9. 数据和外部依赖
10. 权限、安全和隐私
11. 异常、边界和恢复场景
12. MVP 范围
13. 后续候选范围
14. 每项需求的验收标准
15. 尚未确认的问题

每项内容标记为：
- 已确认
- 建议
- 待确认
- 明确不做

不得把你的建议写成我的既定需求。完成后等待我逐项确认。
```

PRD 确认前，至少检查：

- 是否有清晰的目标与非目标？
- 是否能判断项目什么时候算完成？
- 是否覆盖失败和恢复流程？
- 是否存在相互冲突的需求？
- 是否把未来设想误放进 MVP？

---

## 七、第五阶段：架构设计与反方评审

PRD 确认后再设计架构：

```text
PRD 已确认。请提出架构方案，暂时不要编码。

请输出：

1. 系统边界和模块职责
2. 核心数据流
3. 数据模型
4. 接口和外部依赖
5. 权限与安全边界
6. 错误处理和恢复机制
7. 日志、监控和审计
8. 测试策略
9. 部署、升级和回滚方案
10. 至少两个有实质差异的备选方案
11. 各方案的成本、限制、风险和适用条件
12. 推荐方案及理由

然后进行一次反方评审：
- 这个方案最可能如何失败？
- 哪些假设没有证据？
- 是否存在更简单的方案？
- 是否为了未来假设而过度设计？
- 一年后最可能因为什么返工？

对影响较大的架构决定，等待我确认后再写入既定路线。
```

架构评审不能只判断“能不能做”，还要判断：

- 是否解决真实问题？
- 是否便于验证？
- 是否容易回退？
- 是否引入不必要复杂度？
- 是否能在 MVP 的时间和资源内完成？

---

## 八、第六阶段：制定分阶段计划

架构确认后，让 Codex把项目拆成可验收的小里程碑：

```text
请根据已确认的 PRD 和架构制定实施计划。

要求：

1. 每个里程碑都必须能独立验证和回退。
2. 每个里程碑说明目标、非目标、依赖、改动范围、风险和交付物。
3. 每项任务都有明确验收标准。
4. 同时规划正常、边界、失败和安全测试。
5. 优先验证最高风险假设，不要先堆大量低风险功能。
6. 明确哪些任务需要我进行业务确认。
7. 不要把长期候选能力混入当前 MVP。
8. 说明整体完成定义 Definition of Done。

输出：
- 阶段顺序
- 每阶段任务
- 验收标准
- 测试矩阵
- 风险与缓解措施
- 回滚方式
- 用户检查点
```

好的计划不是任务越多越专业，而是能尽早暴露错误，并让每一步都可验证、可停止、可调整。

---

## 九、第七阶段：建立项目持久化文档

正式编码前，要求创建或确认以下文件：

```text
现在建立项目的持久化协作文件：

- AGENTS.md：Codex 必须长期遵守的工程规则、命令和完成标准
- docs/PRD.md：已确认的需求、目标、非目标和验收标准
- docs/ARCHITECTURE.md：架构、模块边界和数据流
- docs/PLAN.md：阶段、任务、状态和依赖
- docs/DECISIONS.md：关键决策、备选方案和选择原因
- docs/TESTING.md：测试策略、命令和通过标准
- docs/CHANGELOG.md：重要行为变化

要求：

1. 已确认的事实和决定可以作为执行基线。
2. 建议和候选想法可以记录，但必须明确标记为未确认，不能作为执行基线。
3. 文档之间不能互相冲突。
4. 明确规定需求变化后应更新哪些文件和测试。
5. 未经我确认，不得改变项目最终目标和关键架构。
```

重要信息的归属：

| 内容 | 应写入 |
| --- | --- |
| 产品目标、范围、验收标准 | `docs/PRD.md` |
| 架构和模块边界 | `docs/ARCHITECTURE.md` |
| 任务进度和下一步 | `docs/PLAN.md` |
| 为什么选择某个方案 | `docs/DECISIONS.md` |
| 构建、测试和质量要求 | `AGENTS.md`、`docs/TESTING.md` |
| 用户纠正和行为变化 | PRD、决策记录、测试、变更记录 |

---

## 十、第八阶段：按单个里程碑实施

不要发送“请完成整个项目”。每次只让 Codex实施一个里程碑。

```text
实施 docs/PLAN.md 中的下一个已确认里程碑。

开始前先报告：

1. 本次目标
2. 明确不做什么
3. 需求依据
4. 当前假设和不确定项
5. 允许修改的范围
6. 可能受影响的模块
7. 验收和回滚方法

实施规则：

1. 保持改动范围最小。
2. 遵循现有代码结构和项目约定。
3. 不得自行扩大产品范围。
4. 新增行为必须有测试。
5. 覆盖正常、边界和失败场景。
6. 发现需求或架构问题时暂停并报告，不要自行改变路线。
7. 代码、测试和必要文档同步更新。

完成后：

1. 运行测试、lint、类型检查和构建。
2. 对照验收标准逐项验证。
3. 审查完整改动。
4. 报告已完成内容、测试证据、未验证内容和剩余风险。
5. 停止并等待我的业务验收。
```

## 十一、第九阶段：验证不能只看“测试通过”

每个里程碑至少经过四层验证：

1. **代码验证**：语法、类型、单元测试、集成测试。
2. **行为验证**：实际运行后的输入输出是否正确。
3. **需求验证**：是否逐项满足 PRD 验收标准。
4. **用户验收**：真实用户操作后是否解决了问题。

工程、CAD 或机械项目还需要额外区分：

1. 软件是否正常运行。
2. CAD 几何是否正确。
3. 模型是否符合设计意图。
4. 零件是否可制造。
5. 机械设计是否安全可靠。

前两项可以大量自动化；后三项不能因为 AI 生成成功就默认成立。

建议使用：

```text
不要只汇报测试是否通过。请分别说明：

- 哪些结论由自动化测试证明
- 哪些结论由真实运行证明
- 哪些结论只经过静态检查
- 哪些结论需要人工业务验收
- 哪些结论仍然没有证据

如果测试覆盖范围有限，必须明确说明，不能把局部通过描述为整体能力完成。
```

## 十二、第十阶段：反方审查和独立复核

实施完成后，让当前 Codex切换到严格审查模式：

```text
暂时不要继续开发。请以反方审查者身份检查本次全部改动。

重点寻找：

- 与 PRD 不一致
- 未经确认的假设
- 边界条件遗漏
- 数据丢失或状态不一致
- 权限、安全和隐私问题
- 异常恢复问题
- 并发和资源问题
- 兼容性回归
- 测试盲区
- 文档与代码不一致
- 为通过测试而过度拟合样例

先报告问题，按严重程度排序并提供证据。不要先解释成果。
发现问题后，给出最小修复方案和重新验证方法。
```

对于重要版本，最好再开启一个独立线程：

```text
你是独立审查者，不参与原方案设计。

请只根据 PRD、架构、验收标准和当前改动进行审查。
不要默认原实现正确，优先寻找行为错误、需求偏差、安全风险、缺失测试和不可回退的改动。
```

独立审查不能完全消除 AI 错误，但可以减少同一思路反复自证的问题。

---

## 十三、第十一阶段：用户业务验收

Codex完成技术验证后，用户仍需确认：

- 产品是否解决了真实问题？
- 操作流程是否符合预期？
- 输出是否容易理解和使用？
- 是否出现未授权的范围变化？
- 哪些行为需要调整？

发现问题后，不要只说“这里改一下”，应要求形成永久约束：

```text
我对本次结果作出以下纠正：

[填写纠正内容]

请先分析这项纠正影响哪些需求、架构、代码、测试和文档。
不得只修改表面行为。

完成后必须：

1. 更新对应需求或验收标准。
2. 记录变更原因。
3. 添加能够防止同类问题再次出现的回归测试。
4. 检查其他模块是否存在相同错误假设。
5. 重新执行受影响的完整验证。
```

## 十四、第十二阶段：需求变更控制

项目进行中产生新想法时，先评估，不要直接实施：

```text
这是一个候选需求，暂时不要编码，也不要写成既定路线：

[填写新想法]

请分析：

1. 它解决什么真实问题？
2. 是否属于当前 MVP？
3. 与现有目标和非目标是否冲突？
4. 对架构、数据、测试、安全和交付时间有什么影响？
5. 是否有更简单的替代方案？
6. 不做它会有什么后果？
7. 建议现在做、以后做，还是不做？

等待我确认后再更新 PRD 和计划。
```

任何需求变化都应留下：

- 原需求
- 变更内容
- 变更原因
- 影响范围
- 决策结果
- 新验收标准
- 回归测试

---

## 十五、每个里程碑的质量门

每个阶段都要通过以下质量门：

### 1. 需求门

- 目标明确。
- 非目标明确。
- 边界条件明确。
- 待确认问题已经解决或被显式记录。

### 2. 设计门

- 模块边界清楚。
- 数据和接口清楚。
- 权限、安全、异常和恢复方案清楚。
- 关键方案经过反方评审。

### 3. 实现门

- 改动范围符合计划。
- 代码、测试和文档同步。
- 没有未经确认的范围扩大。

### 4. 验证门

- 正常、边界、失败和安全测试完成。
- 实际运行结果已检查。
- 验收标准逐项核对。
- 未验证内容明确列出。

### 5. 交付门

- 完整改动已经审查。
- 用户完成业务验收。
- 变更可追踪、可回退。
- 下一阶段没有继承未说明的风险。

---

## 十六、防止项目跑偏的长期规则

建议把下面内容加入每个项目的 `AGENTS.md`：

```markdown
## Requirement Control

- 不得把推测当作已确认需求。
- 需求不明确且会影响业务、数据、权限、架构或安全时，必须向用户确认。
- 不得在实施一个任务时自行扩展产品范围。
- 每个里程碑开始前必须核对目标、非目标和验收标准。
- 每个里程碑完成后必须逐项验证验收标准。
- 用户纠正必须同步更新需求文档、决策记录或回归测试。
- 代码现状与需求文档冲突时，不得默认代码正确。
- 发现路线偏差时必须停止新增功能并报告。
- 没有证据的能力不得标记为已完成或已支持。

## Research And Evidence

- 重要技术结论优先依据官方文档和可靠的一手资料。
- 明确区分事实、推断、建议和待验证假设。
- 资料可能随版本变化时必须重新核实。

## Completion Standard

- 新增行为必须有测试。
- 必须覆盖正常、边界和失败场景。
- 测试通过不等于需求正确，必须核对 PRD。
- 完成后必须审查完整改动并报告剩余风险。
```

## 十七、新项目开场完整版模板

如果希望一次性把启动规则说清楚，可以直接使用下面的完整模板：

```text
我要启动一个新项目。你先作为产品顾问、领域研究员、架构师和风险审查者工作，暂时不要编码。

你的首要目标不是尽快增加代码，而是保证方向正确、事实可信、实现可验证、改动可回退。

工作规则：

1. 先访谈我，确认真实目标、目标用户、核心场景和成功标准。
2. 不要把我提出的实现方式默认当成最佳方案。
3. 主动指出需求遗漏、边界条件、隐含假设、失败方式和长期风险。
4. 对关键专业知识和技术选型查阅官方及可靠资料，并提供来源。
5. 严格区分已核实事实、专业推断、Codex 建议和待确认假设。
6. 提供有实质差异的候选方案，说明成本、限制、风险和适用条件。
7. 不得自行决定影响产品、业务、数据、权限、架构或安全的模糊需求。
8. 关键问题未确认前，不开始编码。
9. 不得把代码存在、示例通过或文档声明当作能力已经可靠。
10. 没有证据时必须明确说尚未验证。

请依次完成：

第一阶段：需求访谈
- 目标和非目标
- 用户和核心流程
- 功能与非功能需求
- 输入、输出和外部依赖
- 异常、边界、安全和恢复要求
- 成功标准和不可接受的错误

第二阶段：专业调研
- 官方资料和可靠来源
- 关键事实
- 方案比较
- 未验证假设
- 需要原型验证的问题

第三阶段：形成 PRD
- 已确认需求
- 建议
- 待确认事项
- 明确不做
- 每项需求的验收标准

第四阶段：架构设计
- 系统边界、模块、数据流和接口
- 安全、错误处理、日志、部署和回滚
- 备选方案、推荐方案和反方评审

第五阶段：实施计划
- 独立、可验证、可回退的小里程碑
- 每阶段目标、非目标、依赖、风险和验收标准
- 正常、边界、失败和安全测试矩阵

第六阶段：建立持久化文档
- AGENTS.md
- docs/PRD.md
- docs/ARCHITECTURE.md
- docs/PLAN.md
- docs/DECISIONS.md
- docs/TESTING.md

完成以上内容并得到我确认后，才能开始第一个里程碑。
现在请先开始需求访谈，不要直接给最终方案。
```

## 十八、项目进行一部分后发现跑偏怎么办

不要继续增加功能，也不要立即大规模重构。先发送：

```text
这个项目已经进行了一部分，但出现了方向偏移。现在暂停新增功能，重新建立可信基线。

本轮只审计，不修改代码。

请分别还原：

1. 项目最终应该解决什么问题。
2. 当前代码实际上能做什么。
3. 文档声称已经完成什么。
4. 三者之间有什么差异。

将内容分类为：
- 已实现且有当前验证
- 已实现但只有历史验证
- 部分实现
- 只有规划
- 已偏离路线
- 无法确认

找出所有未经我确认却被当成需求或架构决定的假设。
检查安全、版本管理、环境、测试、回滚、真实端到端闭环和验收标准。

严格区分事实、推断、建议和待确认假设。
先输出问题和恢复计划，等待我确认后再实施。
```

纠偏的正确顺序是：

1. 暂停新增功能。
2. 还原真实状态。
3. 对齐目标、代码和文档。
4. 修复安全和数据风险。
5. 建立当前环境可信基线。
6. 添加防止同类偏差的规则和测试。
7. 再继续后续里程碑。

## 十九、用户需要亲自控制的事项

即使 Codex承担大部分工作，以下内容仍应由用户最终确认：

- 项目真正要解决的问题。
- 目标用户和优先级。
- MVP 范围和非目标。
- 关键业务规则。
- 风险是否可以接受。
- 重要架构取舍。
- 用户体验是否符合预期。
- 工程、安全、法律或商业上的最终责任。

用户不需要逐行指导代码，但需要在关键检查点确认方向。

## 二十、最简执行清单

每次启动新项目，按下面执行：

- [ ] 先说明协作规则，不直接编码。
- [ ] 让 Codex进行需求访谈。
- [ ] 对关键专业问题进行可靠调研。
- [ ] 确认 PRD、目标、非目标和验收标准。
- [ ] 比较架构方案并进行反方评审。

- [ ] 建立 `AGENTS.md` 和项目文档。
- [ ] 一次只实施一个里程碑。
- [ ] 同时测试正常、边界、失败和安全场景。
- [ ] 检查真实运行，不只看测试数字。
- [ ] 使用独立审查寻找错误。
- [ ] 用户完成业务验收。
- [ ] 每次纠正进入需求、决策记录和回归测试。
- [ ] 出现跑偏时立即暂停、审计、重新建立基线。

## 结论

高效使用 Codex 的关键，不是写出一条完美提示词，也不是要求它“更认真”。

真正可靠的方法是：

```text
需求访谈
→ 专业调研
→ PRD 和验收标准
→ 架构与反方评审
→ 小步计划
→ 受控实施
→ 自动化与真实验证
→ 独立审查
→ 用户验收
→ 纠正沉淀
```

把 Codex 当作一个能力很强、会主动工作的技术团队，同时用文档、证据、测试、审查和用户决策约束它。这样才能把 AI 的能力稳定地转化为专业、严谨、可持续的项目结果。
